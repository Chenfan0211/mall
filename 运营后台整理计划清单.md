# 运营后台整理计划清单

> 本文件记录运营后台方案讨论中已经拍板的结论。
> 每次拍板追加一个版本：v1、v2、v3 依次递增。
> 未拍板内容不写入定案，只在后续讨论确认后追加新版本。

## v1

| 项目 | 拍板结论 |
|------|----------|
| 拍板日期 | 2026-06-18 |
| 方案选择 | 采用 B 方案：按真实运营链路重组运营后台菜单 |
| 一级菜单数量 | 8 个 |
| 供应商/采购归属 | 放进运营后台 |

### 一级菜单定案

| 顺序 | 一级菜单 | 二级页面方向 | 定位 |
|------|----------|--------------|------|
| 1 | 运营驾驶舱 | 今日经营、待办任务、异常预警、城市排行 | 运营一进后台先看 GMV、订单、履约、缺货、退款和待处理任务 |
| 2 | 商品与团期 | 商品中心、类目中心、团期发布、商品白名单、榜单标签 | 把商品发布升级成社区团购的团期经营 |
| 3 | 交易履约 | 订单中心、售后中心、履约跟踪、缺货处理 | 管理订单从支付到自提完成的运营链路 |
| 4 | 自提点/团长 | 自提点管理、团长审核、休假管理、签收/缺货、团长评分 | 将社区团购核心角色独立成运营模块 |
| 5 | 供应商/采购 | 供应商管理、采购协同、送货单、供应商售后、供应商结算 | 供应商和采购协同进入运营后台，不只放在 WMS 或供应商端 |
| 6 | 财务风控 | 分账中心、佣金中心、提现审核、支付退款对账、异常资金 | 资金动作统一收口，降低分账、提现、退款风险 |
| 7 | 用户与营销 | 用户中心、积分中心、评论中心、优惠券、活动页、消息触达 | 用户运营、积分、评论和营销活动统一管理 |
| 8 | 报表与系统 | 报表中心、权限中心、基础配置、系统任务、操作日志 | 管理配置、权限、系统任务和数据分析收口 |

### 第一版新增页面方向

| 页面 | 核心内容 |
|------|----------|
| 运营驾驶舱 | 今日 GMV、订单数、退款率、缺货率、自提点履约率、待处理任务 |
| 团期管理 | 截团时间、配送日、商品池、采购仓、城市/自提点范围、发布前校验 |
| 自提点/团长管理 | 审核、休假、签收率、缺货报备、佣金、评分、异常记录 |
| 异常中心 | 库存不足、配送延误、退款失败、提现异常、分账失败、刷券风险、低评分评论 |
| 对账中心 | 支付对账、退款对账、提现对账、分账对账、异常差异处理 |

### 风控要求

- 财务、退款、提现、分账、库存缺货等动作必须有二次确认。
- 关键动作必须记录操作人、操作时间、处理原因、影响对象和操作日志。
- 异常中心不能只做系统日志，要面向运营展示可处理的业务异常。

| 优先级 | 页面/模块 | 处理方式 |

| P0 | 团期管理 | 新增，并承接现在的商品发布能力 |
| P0 | 异常中心 | 新增，统一处理缺货、退款、提现、分账、配送异常 |

| P0 | 对账中心 | 新增，放到财务风控下面 |
| P1 | 供应商/采购 | 新增一级菜单和页面，供应商资料、采购协同、送货单先做基础版 |
| P1 | 商品中心、类目中心 | 保留，移动到商品与团期 |
| P1 | 订单中心、售后中心 | 保留，移动到交易履约 |
| P1 | 分账中心、佣金中心 | 保留，移动到财务风控 |
| P1 | 用户中心、积分中心、评论中心、营销中心 | 保留，移动到用户与营销 |
| P2 | 报表中心、权限中心、系统任务、基础配置 | 保留，归到报表与系统 |
| 暂缓 | 高级活动编排、复杂 BI、多级审批流配置器 | 第一版先不做，避免后台过重 |

### 落地规则

- 第一轮 UI 只做 P0 页面和现有页面迁移。
- P1 页面先调整菜单归属和入口，不强行做复杂新交互。
- P2 页面只做收口，不作为第一轮重点。
- 商品发布改名为团期管理，但保留商品发布能力。
- 异常中心必须是业务处理台，不是系统日志页。

### 风控优先级

- 异常中心和对账中心必须进入 P0。
- 如果第一轮时间有限，优先保证资金异常、退款异常、提现异常、分账异常、缺货异常、配送延误能被运营看到并处理。
- 系统任务和操作日志只作为辅助证据，不替代异常中心。

## v3

| 项目 | 拍板结论 |
|------|----------|
| 拍板日期 | 2026-06-18 |
| 拍板主题 | 运营驾驶舱规则 |
| 页面定位 | 运营驾驶舱作为运营后台默认首页，面向平台管理员和城市运营展示经营、履约、异常、待办汇总 |

### 驾驶舱区块定案

| 区块 | 拍板规则 |
|------|----------|
| 顶部核心指标 | 固定 6 个：今日 GMV、今日订单数、退款率、缺货率、自提点履约率、待处理异常数 |
| 趋势指标 | 展示 GMV、订单数、退款率、缺货率的 7 日趋势 |
| 待办任务 | 按优先级展示：待审核团期、待处理售后、待审核提现、待处理缺货、待确认采购、待处理异常 |
| 异常预警 | 展示库存不足、配送延误、退款失败、提现异常、分账失败、低评分评论 |
| 城市排行 | 展示城市 GMV、订单数、履约率、退款率、缺货率 |
| 自提点排行 | 展示高 GMV 自提点、低履约自提点、缺货高发自提点 |
| 快捷入口 | 团期管理、订单中心、售后中心、异常中心、对账中心、自提点/团长管理 |
| 默认筛选 | 默认看今日和当前用户数据权限范围 |
| 权限差异 | 平台管理员看全局，城市运营看本城市及下级，供应商/自提点不进入运营驾驶舱 |
| 刷新规则 | 首页指标支持手动刷新，关键异常 1 分钟内刷新或轮询 |
| 空状态 | 无异常时展示“暂无待处理异常”，不要展示空表格 |
| 红色预警 | 退款率、缺货率、履约率、提现异常、分账异常超过阈值时标红 |

### 交互规则

- 运营驾驶舱所有卡片都能点击跳转到对应模块。
- 首页必须有待办任务和异常预警，不能只做数据看板。
- 顶部只放 6 个核心指标，不堆过多数字。
- 驾驶舱不展示供应商端、自提点端、仓库端专属明细，只展示运营视角汇总。
- 城市运营只看本城市及下级数据，不能看全平台。
- 财务异常、履约异常、缺货异常优先级高于普通增长指标。

### 风控要求

- 每个异常必须能跳转到可处理页面。
- 驾驶舱不能只做“好看但不能处理问题”的数据大屏。
- 财务、履约、库存类异常必须展示责任模块、异常对象、发生时间、当前状态和处理入口。

## v4

| 项目 | 拍板结论 |
|------|----------|
| 拍板日期 | 2026-06-18 |
| 拍板主题 | 风控与审批规则 |
| 执行原则 | 高风险动作要审批、二次确认或留痕；普通查询、保存草稿、筛选、查看详情不需要审批 |

### 风控对象定案

| 风控对象 | 拍板规则 |
|----------|----------|
| 商品发布/团期发布 | 发布前必须校验销售时间、截团时间、配送日、采购仓、城市/自提点范围、价格、库存规格绑定 |
| 商品下架/停用 | 必须二次确认，已生成订单不受影响，前台不可继续购买 |
| 订单取消 | 未支付可直接取消；已支付未发货允许运营后台配置“取消订单”功能权限，取消后走退款取消；已发货禁止取消，只能售后 |
| 售后审核 | 通过/驳回都必须填写原因，退款金额不能超过可退金额 |
| 退货退款 | 必须先确认退货，再发起退款 |
| 退款操作 | 必须记录退款单、支付流水、操作人、失败原因，失败可重试 |
| 提现审核 | 不能自动通过，必须财务审核；通过后进入打款流程，驳回释放冻结金额 |
| 分账入账 | 订单完成后才允许生成分账，不允许支付后直接分账 |
| 佣金重算 | 必须保留旧版本，写重算原因，不能覆盖原佣金记录 |
| 对账差异 | 支付、退款、提现、分账差异必须进入异常中心，不能只在报表里展示 |
| 缺货处理 | 必须关联订单、商品、自提点、仓库和责任环节，可触发退款或补发 |
| 库存售卖模式 | 商品/团期支持配置售卖虚拟库存或售卖仓库库存；虚拟库存用于预售集采，仓库库存用于现货售卖；两种模式必须在发布前明确配置 |
| 库存调整 | 报损、盘点差异、手工调整必须审批后生效，并写库存流水；如果商品使用仓库库存售卖，库存调整会影响可售库存；如果使用虚拟库存售卖，需影响虚拟可售量和采购建议量 |
| 自提点休假 | 自提点只能按团期休假；休假团期内小程序端不允许继续下单；已有订单必须继续履约或由运营安排转移处理 |
| 团长/自提点禁用 | 必须确认影响范围，不能影响已支付订单履约 |
| 评论删除 | 必须二次确认，填写原因，保留删除前内容摘要和操作日志 |
| 优惠券停用 | 未使用券不可再用，已使用订单不受影响，需记录停用原因 |
| 权限变更 | 角色、菜单、按钮、数据范围变更必须记录操作日志，建议二次确认 |
| 导出数据 | 涉及用户、订单、财务、手机号时必须受权限控制，并记录导出日志 |

### 拍板原则

- 资金类动作必须审批：提现、退款、分账、佣金重算、对账差异处理。
- 库存类动作必须留流水：缺货、报损、盘点差异、手工调整、退货入库。
- 售卖库存模式必须在商品或团期发布前明确：虚拟库存或仓库库存。
- 影响前台展示的动作必须二次确认：发布、下架、停用、评论删除、优惠券停用。
- 已支付未发货订单，运营后台可通过功能权限执行取消，取消后进入退款取消链路。
- 已发货订单不能后台强制取消，只能走售后。
- 自提点休假按团期控制，休假团期内小程序端不可下单，已有订单继续履约。
- 所有高风险动作必须记录：操作人、操作时间、处理原因、影响对象、前后状态。
- 异常中心作为风控处理入口，对账中心作为资金核对入口，系统日志只做审计证据。
- 普通查询、保存草稿、筛选、查看详情不需要审批。

### 风控提醒

- 虚拟库存可以支撑预售集采，但必须和采购建议量、截团后采购、缺货退款联动，否则容易形成超卖和履约延迟。

## v5

| 项目 | 拍板结论 |
|------|----------|
| 拍板日期 | 2026-06-18 |
| 拍板主题 | 权限与数据隔离规则 |
| 执行原则 | 谁能看、谁能点、谁能导出、谁能审批必须分开控制；菜单可见不等于按钮可操作 |

### 角色边界定案

| 角色 | 数据范围 | 可用能力 | 限制规则 |
|------|----------|----------|----------|
| 平台管理员 | 全平台、全城市、全供应商、全仓库、全自提点 | 全模块查看、配置、审批、导出、操作日志审计 | 高风险动作仍需留痕，不能绕过日志和审批记录 |
| 城市运营 | 当前城市及下级城市、自提点、订单、团期、异常 | 管理城市商品团期、订单履约、售后、自提点、异常处理 | 不能修改平台级财务规则、全局权限规则、跨城市数据 |
| 采购员 | 负责的供应商、采购单、送货单、采购商品、入库协同数据 | 供应商资料查看、采购协同、送货单跟进、采购异常处理 | 不允许导出用户隐私、财务明细、非负责供应商数据 |
| 财务 | 支付、退款、提现、分账、佣金、对账、资金异常 | 财务审核、对账处理、提现处理、退款复核、分账/佣金查看 | 不参与商品发布、团期发布、库存调整等运营动作，除非单独授权只读 |
| 客服/售后 | 订单、售后、用户脱敏信息、售后处理记录 | 售后审核、退款申请发起、用户订单查询、售后沟通记录 | 不允许做财务终审、提现审批、佣金调整、库存调整 |
| 仓库角色 | 所属仓库、库区、库存、入库、出库、盘点、配送任务 | 入库确认、出库处理、库存流水查看、盘点执行、配送任务处理 | 不允许查看全局经营数据、财务数据、非所属仓库数据 |
| 供应商 | 自身供应商资料、商品、采购单、送货单、结算、售后相关数据 | 查看销售、采购、送货、售后、结算状态 | 不进入全局运营驾驶舱，不查看其他供应商和平台全局数据 |
| 自提点/团长 | 自身自提点订单、签收、缺货、佣金、休假、评分数据 | 查看自提点经营、订单签收、休假申请/记录、佣金明细 | 不进入全局运营驾驶舱，不查看其他自提点和城市全局数据 |

### 数据隔离维度

| 维度 | 拍板规则 |
|------|----------|
| 城市维度 | 平台管理员看全局，城市运营只看当前城市及下级城市 |
| 自提点维度 | 自提点/团长只看自身自提点数据，城市运营按城市范围查看 |
| 供应商维度 | 供应商只看自身数据，采购员只看负责供应商，平台管理员可看全部 |
| 仓库维度 | 仓库角色只看所属仓库，跨仓查看和调拨必须单独授权 |
| 角色维度 | 不同角色默认进入不同首页和菜单范围，供应商、自提点、仓库角色不进入运营驾驶舱 |
| 菜单维度 | 一级菜单、二级页面按角色配置可见范围 |
| 按钮维度 | 新增、编辑、删除、审批、取消、导出、重算、退款、调整库存等按钮必须独立授权 |
| 数据范围维度 | 同一个菜单下，不同角色看到的数据范围不同，不能只靠前端隐藏 |

### 敏感信息规则

| 信息类型 | 默认展示 | 完整查看/导出规则 |
|----------|----------|------------------|
| 手机号 | 默认脱敏展示 | 查看完整手机号需要独立权限，并记录查看日志 |
| 收货人/提货人 | 默认可展示姓名，手机号脱敏 | 批量导出完整信息需要导出权限和导出日志 |
| 地址/自提信息 | 按履约需要展示必要信息 | 跨城市、跨自提点查看必须受数据范围限制 |
| 财务数据 | 仅财务、管理员和被授权角色可见 | 导出支付、退款、提现、分账、佣金数据必须记录导出原因 |
| 用户数据 | 客服和运营默认看脱敏信息 | 查看完整用户资料、批量导出用户数据必须单独授权 |

### 按钮权限定案

- 菜单可见不代表可以操作，所有关键按钮必须单独配置权限。
- 订单取消、退款、售后审核、提现审核、分账处理、佣金重算、库存调整、商品下架、团期发布、评论删除、优惠券停用、数据导出必须做按钮级权限。
- 查询、筛选、查看详情可作为基础权限，但跨数据范围查询必须受数据隔离控制。
- 财务按钮权限高于普通运营权限，库存按钮权限高于普通商品权限。
- 已支付未发货订单取消必须依赖“订单取消”按钮权限，且进入退款取消链路。

### 导出与日志规则

- 所有涉及用户、订单、手机号、地址、财务、佣金、供应商结算的数据导出，必须有独立导出权限。
- 导出必须记录：操作人、操作时间、导出模块、导出范围、筛选条件、导出原因、导出文件标识。
- 查看完整手机号、跨城市查询、跨仓库查询、跨供应商查看、代操作都必须记录操作日志。
- 高风险操作日志必须记录前后状态、影响对象、处理原因和审批结果。
- 操作日志只做审计证据，不能替代异常中心和审批流程。

### 默认原则

- 查询权限最小化，默认只看本角色、本城市、本仓库、本供应商或本自提点范围内数据。
- 操作权限按钮化，不能只用菜单权限粗放控制。
- 导出权限单独化，导出比查看权限更高。
- 财务和库存权限要高于普通读写权限。
- 跨角色、跨城市、跨仓库、跨供应商操作必须留痕。
- 供应商、自提点、仓库角色不看平台全局经营数据。

### 下单规则

- 只有“已发布”且未过期的团期允许下单。
- 用户必须在销售开始时间到截团时间之间下单。
- 自提点在该团期休假时，小程序端不允许下单。
- 下单时必须生成商品快照：商品名、规格、价格、库存售卖模式、团期、生产日期、保质期、供应商、采购仓/履约仓。

### 取消与售后规则

- 未支付订单只允许用户取消，运营后台不提供手动取消按钮。
- 未支付订单 15 分钟超时关闭属于系统自动关闭，不属于运营后台手动取消。
- 已支付未履约订单，运营后台可配置“取消订单”功能权限，取消后进入退款取消流程。
- 已发货、待自提、部分自提订单不允许直接取消。
- 只有订单主表进入“已完成”后，才允许发起售后。
- 团期下架不影响已支付订单继续履约。

### 截团后履约规则

- 截团后，系统按团期汇总订单，在仓库后台展示采购单、出库单、司机配送数据。

- 仓库库存模式：按仓库可售库存生成出库单和司机配送数据。
- 所有采购单生成时，必须按“同一配送日 + 同一 SKU”扣减已生成或已占用的采购数量，避免重复采购。
- 缺货时进入异常中心，由运营发起补货，采购再生成补货采购单。

### 自提规则

- 商品到达自提点后，对应订单明细进入待自提。
- 用户提货后，对应订单明细进入已自提。
- 订单下所有商品都已自提后，订单主表才进入已完成。
- 自提点休假只影响新下单，不自动取消已有订单。
- 自提点签收异常、缺货、少货必须进入异常中心。

### 分佣与分账规则

- 分佣按订单商品维度计算，订单详情表可以先进行商品级分佣。
- 订单主表未完成前，不生成最终分账。
- 订单主表已完成后，可生成最终分账。
- 有售后中的订单不影响分账。
- 售后退款后，按退款商品和退款金额扣除各方佣金。
- 自提点、供应商、团长等参与分佣方都需要设置押金，用于抵扣退款后的佣金倒扣，避免佣金账户出现负数。

- 支持按团期、城市、自提点、供应商、仓库、交易状态、支付状态、履约状态筛选。
- 支持查看订单详情、商品快照、支付流水、退款记录、履约记录、分佣记录、操作日志。
- 订单取消、退款、导出、改履约信息必须走按钮权限。
- 涉及手机号、地址、财务金额的导出必须记录导出日志。

- 采购数量允许人工调整，但必须填写调整原因，并记录调整前后数量。
- 供应商创建采购单时，只能填写申请采购数量，不能修改平台系统生成的采购数量。

- 待采购审核。
- 审核驳回。
- 待确认。
- 已确认。
- 待送货。
- 已送货。
- 部分入库。
- 已入库。
- 已取消。
- 异常。

- 供应商只能为自己名下商品/SKU 创建采购单。
- 供应商创建的采购单默认状态为“待采购审核”。

- 采购审核驳回后，采购单作废或退回供应商修改。
- 审核通过前，不允许生成送货单，不允许进入仓库入库流程，不影响采购数量占用。
- 供应商不能审核自己的采购单。
- 采购审核必须记录审核人、审核时间、审核结果、审核意见。

### 供应商协同规则

- 供应商端只能看到自己的采购单、送货单、结算和售后相关数据。
- 供应商需要确认采购单，可填写预计送货时间、可供数量、缺货说明。
- 供应商确认不足量时，采购单进入异常，运营决定补货、换供应商、退款或继续等待。
- 供应商送货时生成送货单，送货单进入仓库入库流程。
- 供应商可查看自己创建的采购单和审核结果。

### 送货与入库规则

- 供应商送货单必须关联采购单。
- 仓库按送货单收货，支持正常入库、部分入库、拒收入库、异常入库。
- 入库差异必须记录：少货、多货、破损、过期、规格不符。
- 入库完成后更新仓库库存和库存流水。
- 虚拟库存模式下，采购入库后再进入后续分拣/配送。

### 采购异常规则

- 供应商缺货、延迟送货、入库差异、质量问题、价格异常，都进入异常中心。
- 异常必须关联采购单、送货单、供应商、SKU、团期、配送日。
- 异常处理结果包括：补货、换供应商、部分履约、退款、关闭异常。
- 采购异常不能只停留在采购页面，必须能在运营驾驶舱和异常中心看到。

### 价格与结算规则

- 采购单记录采购价。
- 采购价变更需要权限和操作日志。
- 最终结算以实际入库数量和确认采购价为准。
- 多送、少送、破损、拒收不直接结算，必须按入库结果处理。

### 权限与日志规则

- 采购员可创建、确认、审核采购单，但不能修改财务结算结果。
- 供应商可创建采购单、查看自己创建的采购单、提交审核。
- 供应商不能审核自己的采购单。
- 供应商不能修改平台系统生成的采购数量，只能对自己创建的采购单填写申请数量。
- 采购数量调整、采购价调整、取消采购单、采购审核必须记录操作日志。
- 涉及供应商结算的数据导出需要独立权限。

### 仓库后台定位

- 仓库后台承接采购入库、库存管理、出库分拣、司机配送、自提点签收。
- 运营后台看结果和异常，仓库后台处理实操任务。
- 仓库角色只能看所属仓库数据。

### 入库规则

- 入库来源包括：采购单送货入库、补货采购入库、退货入库、盘点调整入库。
- 供应商送货单到仓后，仓库按送货单收货。
- 支持正常入库、部分入库、拒收入库、异常入库。
- 入库差异必须记录少货、多货、破损、过期、规格不符。
- 入库完成后更新 SKU 在库数量和库存流水。

### 库存规则

- 库存按 SKU、仓库、批次、生产日期、保质期管理。
- 可售库存、锁定库存、在库库存、异常库存分开。
- 仓库库存模式售卖时，下单或截团后锁定可售库存。
- 虚拟库存模式不直接占用仓库库存，采购入库后再进入分拣配送。
- 库存调整必须写库存流水，盘点差异审核后生效。

### 出库规则

- 仓库库存模式：截团后按团期、配送日、自提点生成出库单。
- 虚拟库存模式：采购入库后按团期、配送日、自提点生成出库单。
- 出库按 FIFO 扣减库存。
- 出库单状态：待分拣、分拣中、待复核、待装车、已出库、异常。
- 少货、破损、错拣必须进入异常中心。

### 司机配送规则

- 司机配送数据在截团时生成，按配送日、线路、自提点、司机进行分配。
- 出库未完成前，司机可以查看配送任务，但不能打卡发车。
- 出库完成后，司机才允许打卡发车。
- 司机打卡发车后，配送任务进入配送中。
- 司机可查看待出库、待发车、配送中、已送达、异常。
- 配送异常包括延迟、错送、拒收、车辆异常。

### 自提点签收规则

- 司机送达自提点后，司机签到即默认自提点签收。
- 司机签到后，对应订单明细进入待自提。
- 自提点第一版不再单独做强制确认签收，以司机到点签到作为签收节点。
- 司机签到默认签收后，如果用户发现缺货、破损、错货等问题，由用户端发起售后处理。
- 运营后台通过售后单、订单明细和配送记录追踪问题，不再要求司机或自提点在签收环节提交缺货、破损异常。

### 库存异常规则

- 缺货、超卖、破损、过期、盘点差异、错拣、错发、拒收都进入异常中心。

- 异常处理方式包括补货、退款、换货、报损、重新配送、关闭异常。

- 入库、出库、盘点、库存调整、报损都要留操作日志。

### 售后发起规则

- 订单商品已收货后，用户端即可针对该订单商品发起售后。
- 售后按订单商品维度发起，不按整单强制处理。
- 用户可针对缺货、破损、错货、质量问题、少件、过期、未提货等原因发起售后。
- 已退款或已关闭售后的订单商品，允许再次发起售后。
- 退款待审核、售后审核中、退款中、补货处理中等售后进行中状态，不允许同一订单商品同时再次发起售后。
- 超出售后期后，不允许再发起售后。

### 售后类型

| 售后类型 | 使用场景 | 拍板规则 |
|----------|----------|----------|
| 仅退款 | 缺货、少件、破损、质量问题、过期、无法履约 | 审核通过后直接进入退款流程 |
| 退货退款 | 需要退回商品后再退款的场景 | 必须确认退货后再退款 |
| 补货/补发 | 可重新履约的缺货、错货、少件 | 平台直接生成补货单处理 |

- 换货第一版不做独立流程，平台按补货单处理。
- 平台补偿第一版暂不做。

### 售后审核规则

- 售后必须由运营/客服审核。
- 审核通过、驳回都必须填写原因。
- 退款金额不能超过该订单商品可退金额。
- 支持部分退款。
- 涉及供应商责任、仓库责任、司机配送责任、自提点责任的售后，需要标记责任方。
- 高金额退款或批量退款第一版暂不要求财务复核。

### 退款规则

- 退款必须关联原支付流水、订单商品、售后单。
- 退款失败进入异常中心，可重试。
- 部分退款后，订单支付状态进入部分退款。
- 全额退款后，对应订单商品进入已退款。
- 退款完成后必须反向扣除相关方佣金。

### 分佣扣回与押金规则

- 售后中的订单不影响已生成分账。
- 售后退款完成后，按退款商品和退款金额扣回自提点、团长、供应商等相关方佣金。
- 押金是各方余额的一部分，但押金不可提现。
- 只有售后扣回场景允许各方佣金余额变为负数。
- 提现不能让佣金余额变为负数。
- 押金不足或佣金倒扣后余额为负时，进入资金异常，后续收益优先冲抵负数。
- 除非解除合同并手动释放押金，否则押金不允许提走。
- 佣金扣回必须保留原分佣记录和扣回记录，不能覆盖原记录。

### 退货入库规则

- 退货退款需要仓库或自提点确认退货后，再进入退款流程。
- 退货商品按实际情况进入可售库存、异常库存或报损。
- 退货入库必须写库存流水。
- 生鲜、临期、破损、过期商品默认不能重新入可售库存。

### 异常联动

- 缺货、破损、错货、少件、退款失败、押金不足、责任方争议进入异常中心。
- 售后单必须关联订单、订单商品、团期、SKU、自提点、供应商、仓库、配送任务。
- 异常中心只处理需要运营介入的问题，普通售后审核仍在售后中心处理。

### 后台能力

- 售后中心支持按团期、城市、自提点、供应商、仓库、售后类型、售后状态、责任方筛选。
- 支持查看订单商品快照、支付流水、退款记录、配送记录、分佣扣回记录。
- 审核、退款、驳回、补货、责任方认定必须做按钮权限。
- 售后导出涉及用户、手机号、金额时必须记录导出日志。

### 核心指标口径

- GMV：已支付订单商品实付金额，不含未支付订单，不含已取消未支付订单。
- 支付金额：实际支付成功金额，以支付流水为准。
- 退款金额：退款成功金额，以退款流水为准。
- 净销售额：支付金额 - 退款成功金额。
- 订单数：按订单主表统计。
- 商品销量：按订单商品明细统计。
- 客单价：支付金额 / 支付订单数。
- 缺货率：缺货订单商品数 / 应履约订单商品数。
- 履约率：已自提订单商品数 / 应自提订单商品数。
- 售后率：发起售后的订单商品数 / 已收货订单商品数。
- 退款率：退款成功金额 / 支付金额。
- 自提点履约率：该自提点已自提订单商品数 / 该自提点应自提订单商品数。
- 供应商缺货率：该供应商缺货订单商品数 / 该供应商应供订单商品数。

### 时间口径

- 经营报表默认按支付时间统计。
- 团期报表按团期截团时间和配送日都可筛选，默认按截团时间。
- 履约报表按履约节点时间统计，例如入库时间、出库时间、司机签到时间、用户自提时间。
- 售后报表按售后申请时间统计，退款金额另支持按退款成功时间。
- 财务报表按资金流水时间统计。
- 报表默认展示今日、近 7 日、近 30 日，可自定义时间范围。

### 权限与导出

- 平台管理员可看全平台报表。
- 城市运营只看当前城市及下级城市。
- 供应商只能看自身供应商相关报表。
- 自提点/团长只能看自身自提点报表。
- 财务可看资金类报表，但不默认开放商品运营编辑权限。
- 所有涉及订单、手机号、用户、财务、佣金、供应商结算的数据导出必须记录导出日志。
- 报表导出必须记录导出人、时间、报表类型、筛选条件、数据范围、导出原因。

### 边界规则

- 报表中心只展示和导出，不直接处理退款、提现、分账、缺货、售后。
- 对账差异去对账中心处理。
- 业务异常去异常中心处理。
- 报表数字允许有 T+1 汇总报表，实时经营指标以运营驾驶舱为准。
- 报表页面必须展示数据更新时间，避免运营误以为都是实时数据。

| 短信 | 第一版不做常规营销短信，只保留后续可扩展位置 |

### 触发场景

| 对象 | 通知场景 |
|------|----------|
| 用户 | 支付成功、支付剩余 5 分钟提醒、自提提醒、售后进度、退款结果、积分/优惠券到期提醒 |
| 运营 | 待审核团期、待处理售后、退款失败、提现审核、分账失败、缺货异常、采购异常 |
| 采购/供应商 | 采购单生成、采购审核结果、供应商确认采购、预计送货提醒、缺货/质量异常 |
| 自提点/团长 | 配送到达、司机签到默认签收、待自提订单汇总、休假审核结果、佣金/提现通知 |
| 财务 | 提现待审、退款失败、对账差异、分账失败、押金不足、佣金为负异常 |

### 发送规则

- 每条消息必须关联业务对象，例如订单、订单商品、团期、采购单、售后单、提现单、异常单。
- 消息必须能跳转到对应详情页或处理页。
- 同一个业务对象、同一个事件不能重复刷消息，必须做幂等。
- 重要通知要保存发送记录：接收人、渠道、模板、业务对象、发送状态、失败原因。
- 外部推送失败，不影响业务状态，但关键业务必须保留后台待办。
- 失败消息可以重试，但重试不能重复改变业务单据状态。
- 营销类消息必须受用户授权、订阅状态和频控限制。

### 边界规则

- 消息中心不替代异常中心，只有关键业务通知失败或影响履约/资金处理时才进入异常中心。
- 消息中心不替代操作日志，操作日志记录谁做了什么，消息记录通知了谁。
- 消息模板归到基础配置，修改规则按 v12 的配置修改、立即生效、逻辑删除、停用、日志规则执行。
- 第一版先保证业务通知和后台待办，不做复杂营销自动化。

- 用户中心只能查看和管理 C 端用户资料，不能把 C 端用户作为后台账号使用。
- 如果手机号同时属于 C 端用户和后台员工，必须通过后台账号体系单独开通，不允许直接用用户身份登录 PC 端。
- 用户端仍然只通过小程序/H5 等用户端入口登录。

### 用户中心规则

- 用户资料默认脱敏展示，手机号完整查看需要独立权限和日志。
- 运营可以查看用户订单、售后、积分、优惠券、评论、风险记录。
- 用户账号冻结、解冻、风险标记必须填写原因并留操作日志。
- 用户中心不能直接改订单金额、退款金额、佣金、提现、分账。
- 批量导出用户数据必须有导出权限和导出原因。

### 积分规则

- 积分必须有账户汇总和流水明细。
- 每条积分流水必须记录来源、变动值、过期时间、关联业务对象。
- 下单返积分按订单商品维度计算，退款后按退款商品扣回对应积分。
- 积分过期由系统任务处理，过期流水必须可追溯。
- 手工增减积分必须审批或二次确认，并填写原因。
- 积分余额不允许因为普通消费变成负数；售后扣回导致不足时进入异常或后续积分优先冲抵。

### 优惠券规则

- 优惠券分为券模板和用户券。
- 券模板配置面额、门槛、有效期、发放数量、适用范围、使用说明。
- 第一版优惠券适用范围先支持全场、类目、商品、团期，不做复杂人群包自动投放。
- 优惠券停用后，未领取不可继续领取，已领取未使用是否可用按停用策略处理。
- 已使用优惠券的订单取消或退款时，按规则决定是否退回券，必须记录退券流水。
- 批量发券、手工补券、作废用户券必须留操作日志。

### 评论规则

- 评论中心第一版不做回复功能。
- 评论只能查看、筛选、隐藏、删除，不能直接篡改用户原文。
- 删除评论必须二次确认，填写原因，并保留删除前内容摘要。
- 低评分评论进入运营待处理，可联动异常中心。
- 评论涉及售后、缺货、配送问题时，要能跳转订单、售后、团期、自提点。
- 恶意评论或异常用户可打风险标签，但不能自动封号，封号需单独操作权限。

### 活动页规则

- 第一版活动页只做基础配置：Banner、入口图、跳转链接、展示时间、启停状态。
- 活动页发布、停用必须二次确认。
- 已发布活动配置修改后立即生效，但必须记录修改前后内容。
- 活动页不能绕过团期上下架、库存、城市权限和商品可售状态。

### 边界规则

- 用户与营销不处理履约、采购、出库、退款、提现、分账。
- 营销动作不能绕过库存、团期、订单、售后规则。
- 涉及用户资产的动作，包括积分、优惠券、补偿类发放，都必须留流水。
- 第一版重点是可查、可控、可追溯，不做复杂营销自动化。
