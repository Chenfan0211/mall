# 运营后台整理计划清单

> 本文件记录运营后台方案讨论中已经拍板的结论。
> 每次拍板追加一个版本：v1、v2、v3 依次递增。
> 未拍板内容不写入定案，只在后续讨论确认后追加新版本。

## v1

| 项目 | 拍板结论 |
|------|----------|
| 拍板日期 | 2026-06-18 |
| 方案选择 | 采用 B 方案：按真实运营链路重组运营后台菜单 |
| 一级菜单数量 | 8 个 |
| 供应商/采购归属 | 放进运营后台 |

### 一级菜单定案

| 顺序 | 一级菜单 | 二级页面方向 | 定位 |
|------|----------|--------------|------|
| 1 | 运营驾驶舱 | 今日经营、待办任务、异常预警、城市排行 | 运营一进后台先看 GMV、订单、履约、缺货、退款和待处理任务 |
| 2 | 商品与团期 | 商品中心、类目中心、团期发布、商品白名单、榜单标签 | 把商品发布升级成社区团购的团期经营 |
| 3 | 交易履约 | 订单中心、售后中心、履约跟踪、缺货处理 | 管理订单从支付到自提完成的运营链路 |
| 4 | 自提点/团长 | 自提点管理、团长审核、休假管理、签收/缺货、团长评分 | 将社区团购核心角色独立成运营模块 |
| 5 | 供应商/采购 | 供应商管理、采购协同、送货单、供应商售后、供应商结算 | 供应商和采购协同进入运营后台，不只放在 WMS 或供应商端 |
| 6 | 财务风控 | 分账中心、佣金中心、提现审核、支付退款对账、异常资金 | 资金动作统一收口，降低分账、提现、退款风险 |
| 7 | 用户与营销 | 用户中心、积分中心、评论中心、优惠券、活动页、消息触达 | 用户运营、积分、评论和营销活动统一管理 |
| 8 | 报表与系统 | 报表中心、权限中心、基础配置、系统任务、操作日志 | 管理配置、权限、系统任务和数据分析收口 |

### 第一版新增页面方向

| 页面 | 核心内容 |
|------|----------|
| 运营驾驶舱 | 今日 GMV、订单数、退款率、缺货率、自提点履约率、待处理任务 |
| 团期管理 | 截团时间、配送日、商品池、采购仓、城市/自提点范围、发布前校验 |
| 自提点/团长管理 | 审核、休假、签收率、缺货报备、佣金、评分、异常记录 |
| 异常中心 | 库存不足、配送延误、退款失败、提现异常、分账失败、刷券风险、低评分评论 |
| 对账中心 | 支付对账、退款对账、提现对账、分账对账、异常差异处理 |

### 风控要求

- 财务、退款、提现、分账、库存缺货等动作必须有二次确认。
- 关键动作必须记录操作人、操作时间、处理原因、影响对象和操作日志。
- 异常中心不能只做系统日志，要面向运营展示可处理的业务异常。

| 优先级 | 页面/模块 | 处理方式 |

| P0 | 团期管理 | 新增，并承接现在的商品发布能力 |
| P0 | 异常中心 | 新增，统一处理缺货、退款、提现、分账、配送异常 |

| P0 | 对账中心 | 新增，放到财务风控下面 |
| P1 | 供应商/采购 | 新增一级菜单和页面，供应商资料、采购协同、送货单先做基础版 |
| P1 | 商品中心、类目中心 | 保留，移动到商品与团期 |
| P1 | 订单中心、售后中心 | 保留，移动到交易履约 |
| P1 | 分账中心、佣金中心 | 保留，移动到财务风控 |
| P1 | 用户中心、积分中心、评论中心、营销中心 | 保留，移动到用户与营销 |
| P2 | 报表中心、权限中心、系统任务、基础配置 | 保留，归到报表与系统 |
| 暂缓 | 高级活动编排、复杂 BI、多级审批流配置器 | 第一版先不做，避免后台过重 |

### 落地规则

- 第一轮 UI 只做 P0 页面和现有页面迁移。
- P1 页面先调整菜单归属和入口，不强行做复杂新交互。
- P2 页面只做收口，不作为第一轮重点。
- 商品发布改名为团期管理，但保留商品发布能力。
- 异常中心必须是业务处理台，不是系统日志页。

### 风控优先级

- 异常中心和对账中心必须进入 P0。
- 如果第一轮时间有限，优先保证资金异常、退款异常、提现异常、分账异常、缺货异常、配送延误能被运营看到并处理。
- 系统任务和操作日志只作为辅助证据，不替代异常中心。

## v3

| 项目 | 拍板结论 |
|------|----------|
| 拍板日期 | 2026-06-18 |
| 拍板主题 | 运营驾驶舱规则 |
| 页面定位 | 运营驾驶舱作为运营后台默认首页，面向平台管理员和城市运营展示经营、履约、异常、待办汇总 |

### 驾驶舱区块定案

| 区块 | 拍板规则 |
|------|----------|
| 顶部核心指标 | 固定 6 个：今日 GMV、今日订单数、退款率、缺货率、自提点履约率、待处理异常数 |
| 趋势指标 | 展示 GMV、订单数、退款率、缺货率的 7 日趋势 |
| 待办任务 | 按优先级展示：待审核团期、待处理售后、待审核提现、待处理缺货、待确认采购、待处理异常 |
| 异常预警 | 展示库存不足、配送延误、退款失败、提现异常、分账失败、低评分评论 |
| 城市排行 | 展示城市 GMV、订单数、履约率、退款率、缺货率 |
| 自提点排行 | 展示高 GMV 自提点、低履约自提点、缺货高发自提点 |
| 快捷入口 | 团期管理、订单中心、售后中心、异常中心、对账中心、自提点/团长管理 |
| 默认筛选 | 默认看今日和当前用户数据权限范围 |
| 权限差异 | 平台管理员看全局，城市运营看本城市及下级，供应商/自提点不进入运营驾驶舱 |
| 刷新规则 | 首页指标支持手动刷新，关键异常 1 分钟内刷新或轮询 |
| 空状态 | 无异常时展示“暂无待处理异常”，不要展示空表格 |
| 红色预警 | 退款率、缺货率、履约率、提现异常、分账异常超过阈值时标红 |

### 交互规则

- 运营驾驶舱所有卡片都能点击跳转到对应模块。
- 首页必须有待办任务和异常预警，不能只做数据看板。
- 顶部只放 6 个核心指标，不堆过多数字。
- 驾驶舱不展示供应商端、自提点端、仓库端专属明细，只展示运营视角汇总。
- 城市运营只看本城市及下级数据，不能看全平台。
- 财务异常、履约异常、缺货异常优先级高于普通增长指标。

### 风控要求

- 每个异常必须能跳转到可处理页面。
- 驾驶舱不能只做“好看但不能处理问题”的数据大屏。
- 财务、履约、库存类异常必须展示责任模块、异常对象、发生时间、当前状态和处理入口。

## v4

| 项目 | 拍板结论 |
|------|----------|
| 拍板日期 | 2026-06-18 |
| 拍板主题 | 风控与审批规则 |
| 执行原则 | 高风险动作要审批、二次确认或留痕；普通查询、保存草稿、筛选、查看详情不需要审批 |

### 风控对象定案

| 风控对象 | 拍板规则 |
|----------|----------|
| 商品发布/团期发布 | 发布前必须校验销售时间、截团时间、配送日、采购仓、城市/自提点范围、价格、库存规格绑定 |
| 商品下架/停用 | 必须二次确认，已生成订单不受影响，前台不可继续购买 |
| 订单取消 | 未支付可直接取消；已支付未发货允许运营后台配置“取消订单”功能权限，取消后走退款取消；已发货禁止取消，只能售后 |
| 售后审核 | 通过/驳回都必须填写原因，退款金额不能超过可退金额 |
| 退货退款 | 必须先确认退货，再发起退款 |
| 退款操作 | 必须记录退款单、支付流水、操作人、失败原因，失败可重试 |
| 提现审核 | 不能自动通过，必须财务审核；通过后进入打款流程，驳回释放冻结金额 |
| 分账入账 | 订单完成后才允许生成分账，不允许支付后直接分账 |
| 佣金重算 | 必须保留旧版本，写重算原因，不能覆盖原佣金记录 |
| 对账差异 | 支付、退款、提现、分账差异必须进入异常中心，不能只在报表里展示 |
| 缺货处理 | 必须关联订单、商品、自提点、仓库和责任环节，可触发退款或补发 |
| 库存售卖模式 | 商品/团期支持配置售卖虚拟库存或售卖仓库库存；虚拟库存用于预售集采，仓库库存用于现货售卖；两种模式必须在发布前明确配置 |
| 库存调整 | 报损、盘点差异、手工调整必须审批后生效，并写库存流水；如果商品使用仓库库存售卖，库存调整会影响可售库存；如果使用虚拟库存售卖，需影响虚拟可售量和采购建议量 |
| 自提点休假 | 自提点只能按团期休假；休假团期内小程序端不允许继续下单；已有订单必须继续履约或由运营安排转移处理 |
| 团长/自提点禁用 | 必须确认影响范围，不能影响已支付订单履约 |
| 评论删除 | 必须二次确认，填写原因，保留删除前内容摘要和操作日志 |
| 优惠券停用 | 未使用券不可再用，已使用订单不受影响，需记录停用原因 |
| 权限变更 | 角色、菜单、按钮、数据范围变更必须记录操作日志，建议二次确认 |
| 导出数据 | 涉及用户、订单、财务、手机号时必须受权限控制，并记录导出日志 |

### 拍板原则

- 资金类动作必须审批：提现、退款、分账、佣金重算、对账差异处理。
- 库存类动作必须留流水：缺货、报损、盘点差异、手工调整、退货入库。
- 售卖库存模式必须在商品或团期发布前明确：虚拟库存或仓库库存。
- 影响前台展示的动作必须二次确认：发布、下架、停用、评论删除、优惠券停用。
- 已支付未发货订单，运营后台可通过功能权限执行取消，取消后进入退款取消链路。
- 已发货订单不能后台强制取消，只能走售后。
- 自提点休假按团期控制，休假团期内小程序端不可下单，已有订单继续履约。
- 所有高风险动作必须记录：操作人、操作时间、处理原因、影响对象、前后状态。
- 异常中心作为风控处理入口，对账中心作为资金核对入口，系统日志只做审计证据。
- 普通查询、保存草稿、筛选、查看详情不需要审批。

### 风控提醒

- 虚拟库存可以支撑预售集采，但必须和采购建议量、截团后采购、缺货退款联动，否则容易形成超卖和履约延迟。

## v5

| 项目 | 拍板结论 |
|------|----------|
| 拍板日期 | 2026-06-18 |
| 拍板主题 | 权限与数据隔离规则 |
| 执行原则 | 谁能看、谁能点、谁能导出、谁能审批必须分开控制；菜单可见不等于按钮可操作 |

### 角色边界定案

| 角色 | 数据范围 | 可用能力 | 限制规则 |
|------|----------|----------|----------|
| 平台管理员 | 全平台、全城市、全供应商、全仓库、全自提点 | 全模块查看、配置、审批、导出、操作日志审计 | 高风险动作仍需留痕，不能绕过日志和审批记录 |
| 城市运营 | 当前城市及下级城市、自提点、订单、团期、异常 | 管理城市商品团期、订单履约、售后、自提点、异常处理 | 不能修改平台级财务规则、全局权限规则、跨城市数据 |
| 采购员 | 负责的供应商、采购单、送货单、采购商品、入库协同数据 | 供应商资料查看、采购协同、送货单跟进、采购异常处理 | 不允许导出用户隐私、财务明细、非负责供应商数据 |
| 财务 | 支付、退款、提现、分账、佣金、对账、资金异常 | 财务审核、对账处理、提现处理、退款复核、分账/佣金查看 | 不参与商品发布、团期发布、库存调整等运营动作，除非单独授权只读 |
| 客服/售后 | 订单、售后、用户脱敏信息、售后处理记录 | 售后审核、退款申请发起、用户订单查询、售后沟通记录 | 不允许做财务终审、提现审批、佣金调整、库存调整 |
| 仓库角色 | 所属仓库、库区、库存、入库、出库、盘点、配送任务 | 入库确认、出库处理、库存流水查看、盘点执行、配送任务处理 | 不允许查看全局经营数据、财务数据、非所属仓库数据 |
| 供应商 | 自身供应商资料、商品、采购单、送货单、结算、售后相关数据 | 查看销售、采购、送货、售后、结算状态 | 不进入全局运营驾驶舱，不查看其他供应商和平台全局数据 |
| 自提点/团长 | 自身自提点订单、签收、缺货、佣金、休假、评分数据 | 查看自提点经营、订单签收、休假申请/记录、佣金明细 | 不进入全局运营驾驶舱，不查看其他自提点和城市全局数据 |

### 数据隔离维度

| 维度 | 拍板规则 |
|------|----------|
| 城市维度 | 平台管理员看全局，城市运营只看当前城市及下级城市 |
| 自提点维度 | 自提点/团长只看自身自提点数据，城市运营按城市范围查看 |
| 供应商维度 | 供应商只看自身数据，采购员只看负责供应商，平台管理员可看全部 |
| 仓库维度 | 仓库角色只看所属仓库，跨仓查看和调拨必须单独授权 |
| 角色维度 | 不同角色默认进入不同首页和菜单范围，供应商、自提点、仓库角色不进入运营驾驶舱 |
| 菜单维度 | 一级菜单、二级页面按角色配置可见范围 |
| 按钮维度 | 新增、编辑、删除、审批、取消、导出、重算、退款、调整库存等按钮必须独立授权 |
| 数据范围维度 | 同一个菜单下，不同角色看到的数据范围不同，不能只靠前端隐藏 |

### 敏感信息规则

| 信息类型 | 默认展示 | 完整查看/导出规则 |
|----------|----------|------------------|
| 手机号 | 默认脱敏展示 | 查看完整手机号需要独立权限，并记录查看日志 |
| 收货人/提货人 | 默认可展示姓名，手机号脱敏 | 批量导出完整信息需要导出权限和导出日志 |
| 地址/自提信息 | 按履约需要展示必要信息 | 跨城市、跨自提点查看必须受数据范围限制 |
| 财务数据 | 仅财务、管理员和被授权角色可见 | 导出支付、退款、提现、分账、佣金数据必须记录导出原因 |
| 用户数据 | 客服和运营默认看脱敏信息 | 查看完整用户资料、批量导出用户数据必须单独授权 |

### 按钮权限定案

- 菜单可见不代表可以操作，所有关键按钮必须单独配置权限。
- 订单取消、退款、售后审核、提现审核、分账处理、佣金重算、库存调整、商品下架、团期发布、评论删除、优惠券停用、数据导出必须做按钮级权限。
- 查询、筛选、查看详情可作为基础权限，但跨数据范围查询必须受数据隔离控制。
- 财务按钮权限高于普通运营权限，库存按钮权限高于普通商品权限。
- 已支付未发货订单取消必须依赖“订单取消”按钮权限，且进入退款取消链路。

### 导出与日志规则

- 所有涉及用户、订单、手机号、地址、财务、佣金、供应商结算的数据导出，必须有独立导出权限。
- 导出必须记录：操作人、操作时间、导出模块、导出范围、筛选条件、导出原因、导出文件标识。
- 查看完整手机号、跨城市查询、跨仓库查询、跨供应商查看、代操作都必须记录操作日志。
- 高风险操作日志必须记录前后状态、影响对象、处理原因和审批结果。
- 操作日志只做审计证据，不能替代异常中心和审批流程。

### 默认原则

- 查询权限最小化，默认只看本角色、本城市、本仓库、本供应商或本自提点范围内数据。
- 操作权限按钮化，不能只用菜单权限粗放控制。
- 导出权限单独化，导出比查看权限更高。
- 财务和库存权限要高于普通读写权限。
- 跨角色、跨城市、跨仓库、跨供应商操作必须留痕。
- 供应商、自提点、仓库角色不看平台全局经营数据。

### 下单规则

- 只有“已发布”且未过期的团期允许下单。
- 用户必须在销售开始时间到截团时间之间下单。
- 自提点在该团期休假时，小程序端不允许下单。
- 下单时必须生成商品快照：商品名、规格、价格、库存售卖模式、团期、生产日期、保质期、供应商、采购仓/履约仓。

### 取消与售后规则

- 未支付订单只允许用户取消，运营后台不提供手动取消按钮。
- 未支付订单 15 分钟超时关闭属于系统自动关闭，不属于运营后台手动取消。
- 已支付未履约订单，运营后台可配置“取消订单”功能权限，取消后进入退款取消流程。
- 已发货、待自提、部分自提订单不允许直接取消。
- 只有订单主表进入“已完成”后，才允许发起售后。
- 团期下架不影响已支付订单继续履约。

### 截团后履约规则

- 截团后，系统按团期汇总订单，在仓库后台展示采购单、出库单、司机配送数据。

- 仓库库存模式：按仓库可售库存生成出库单和司机配送数据。
- 所有采购单生成时，必须按“同一配送日 + 同一 SKU”扣减已生成或已占用的采购数量，避免重复采购。
- 缺货时进入异常中心，由运营发起补货，采购再生成补货采购单。

### 自提规则

- 商品到达自提点后，对应订单明细进入待自提。
- 用户提货后，对应订单明细进入已自提。
- 订单下所有商品都已自提后，订单主表才进入已完成。
- 自提点休假只影响新下单，不自动取消已有订单。
- 自提点签收异常、缺货、少货必须进入异常中心。

### 分佣与分账规则

- 分佣按订单商品维度计算，订单详情表可以先进行商品级分佣。
- 订单主表未完成前，不生成最终分账。
- 订单主表已完成后，可生成最终分账。
- 有售后中的订单不影响分账。
- 售后退款后，按退款商品和退款金额扣除各方佣金。
- 自提点、供应商、团长等参与分佣方都需要设置押金，用于抵扣退款后的佣金倒扣，避免佣金账户出现负数。

- 支持按团期、城市、自提点、供应商、仓库、交易状态、支付状态、履约状态筛选。
- 支持查看订单详情、商品快照、支付流水、退款记录、履约记录、分佣记录、操作日志。
- 订单取消、退款、导出、改履约信息必须走按钮权限。
- 涉及手机号、地址、财务金额的导出必须记录导出日志。

- 采购数量允许人工调整，但必须填写调整原因，并记录调整前后数量。
- 供应商创建采购单时，只能填写申请采购数量，不能修改平台系统生成的采购数量。

- 待采购审核。
- 审核驳回。
- 待确认。
- 已确认。
- 待送货。
- 已送货。
- 部分入库。
- 已入库。
- 已取消。
- 异常。

- 供应商只能为自己名下商品/SKU 创建采购单。
- 供应商创建的采购单默认状态为“待采购审核”。

- 采购审核驳回后，采购单作废或退回供应商修改。
- 审核通过前，不允许生成送货单，不允许进入仓库入库流程，不影响采购数量占用。
- 供应商不能审核自己的采购单。
- 采购审核必须记录审核人、审核时间、审核结果、审核意见。

### 供应商协同规则

- 供应商端只能看到自己的采购单、送货单、结算和售后相关数据。
- 供应商需要确认采购单，可填写预计送货时间、可供数量、缺货说明。
- 供应商确认不足量时，采购单进入异常，运营决定补货、换供应商、退款或继续等待。
- 供应商送货时生成送货单，送货单进入仓库入库流程。
- 供应商可查看自己创建的采购单和审核结果。

### 送货与入库规则

- 供应商送货单必须关联采购单。
- 仓库按送货单收货，支持正常入库、部分入库、拒收入库、异常入库。
- 入库差异必须记录：少货、多货、破损、过期、规格不符。
- 入库完成后更新仓库库存和库存流水。
- 虚拟库存模式下，采购入库后再进入后续分拣/配送。

### 采购异常规则

- 供应商缺货、延迟送货、入库差异、质量问题、价格异常，都进入异常中心。
- 异常必须关联采购单、送货单、供应商、SKU、团期、配送日。
- 异常处理结果包括：补货、换供应商、部分履约、退款、关闭异常。
- 采购异常不能只停留在采购页面，必须能在运营驾驶舱和异常中心看到。

### 价格与结算规则

- 采购单记录采购价。
- 采购价变更需要权限和操作日志。
- 最终结算以实际入库数量和确认采购价为准。
- 多送、少送、破损、拒收不直接结算，必须按入库结果处理。

### 权限与日志规则

- 采购员可创建、确认、审核采购单，但不能修改财务结算结果。
- 供应商可创建采购单、查看自己创建的采购单、提交审核。
- 供应商不能审核自己的采购单。
- 供应商不能修改平台系统生成的采购数量，只能对自己创建的采购单填写申请数量。
- 采购数量调整、采购价调整、取消采购单、采购审核必须记录操作日志。
- 涉及供应商结算的数据导出需要独立权限。

### 仓库后台定位

- 仓库后台承接采购入库、库存管理、出库分拣、司机配送、自提点签收。
- 运营后台看结果和异常，仓库后台处理实操任务。
- 仓库角色只能看所属仓库数据。

### 入库规则

- 入库来源包括：采购单送货入库、补货采购入库、退货入库、盘点调整入库。
- 供应商送货单到仓后，仓库按送货单收货。
- 支持正常入库、部分入库、拒收入库、异常入库。
- 入库差异必须记录少货、多货、破损、过期、规格不符。
- 入库完成后更新 SKU 在库数量和库存流水。

### 库存规则

- 库存按 SKU、仓库、批次、生产日期、保质期管理。
- 可售库存、锁定库存、在库库存、异常库存分开。
- 仓库库存模式售卖时，下单或截团后锁定可售库存。
- 虚拟库存模式不直接占用仓库库存，采购入库后再进入分拣配送。
- 库存调整必须写库存流水，盘点差异审核后生效。

### 出库规则

- 仓库库存模式：截团后按团期、配送日、自提点生成出库单。
- 虚拟库存模式：采购入库后按团期、配送日、自提点生成出库单。
- 出库按 FIFO 扣减库存。
- 出库单状态：待分拣、分拣中、待复核、待装车、已出库、异常。
- 少货、破损、错拣必须进入异常中心。

### 司机配送规则

- 司机配送数据在截团时生成，按配送日、线路、自提点、司机进行分配。
- 出库未完成前，司机可以查看配送任务，但不能打卡发车。
- 出库完成后，司机才允许打卡发车。
- 司机打卡发车后，配送任务进入配送中。
- 司机可查看待出库、待发车、配送中、已送达、异常。
- 配送异常包括延迟、错送、拒收、车辆异常。

### 自提点签收规则

- 司机送达自提点后，司机签到即默认自提点签收。
- 司机签到后，对应订单明细进入待自提。
- 自提点第一版不再单独做强制确认签收，以司机到点签到作为签收节点。
- 司机签到默认签收后，如果用户发现缺货、破损、错货等问题，由用户端发起售后处理。
- 运营后台通过售后单、订单明细和配送记录追踪问题，不再要求司机或自提点在签收环节提交缺货、破损异常。

### 库存异常规则

- 缺货、超卖、破损、过期、盘点差异、错拣、错发、拒收都进入异常中心。

- 异常处理方式包括补货、退款、换货、报损、重新配送、关闭异常。

- 入库、出库、盘点、库存调整、报损都要留操作日志。

### 售后发起规则

- 订单商品已收货后，用户端即可针对该订单商品发起售后。
- 售后按订单商品维度发起，不按整单强制处理。
- 用户可针对缺货、破损、错货、质量问题、少件、过期、未提货等原因发起售后。
- 已退款或已关闭售后的订单商品，允许再次发起售后。
- 退款待审核、售后审核中、退款中、补货处理中等售后进行中状态，不允许同一订单商品同时再次发起售后。
- 超出售后期后，不允许再发起售后。

### 售后类型

| 售后类型 | 使用场景 | 拍板规则 |
|----------|----------|----------|
| 仅退款 | 缺货、少件、破损、质量问题、过期、无法履约 | 审核通过后直接进入退款流程 |
| 退货退款 | 需要退回商品后再退款的场景 | 必须确认退货后再退款 |
| 补货/补发 | 可重新履约的缺货、错货、少件 | 平台直接生成补货单处理 |

- 换货第一版不做独立流程，平台按补货单处理。
- 平台补偿第一版暂不做。

### 售后审核规则

- 售后必须由运营/客服审核。
- 审核通过、驳回都必须填写原因。
- 退款金额不能超过该订单商品可退金额。
- 支持部分退款。
- 涉及供应商责任、仓库责任、司机配送责任、自提点责任的售后，需要标记责任方。
- 高金额退款或批量退款第一版暂不要求财务复核。

### 退款规则

- 退款必须关联原支付流水、订单商品、售后单。
- 退款失败进入异常中心，可重试。
- 部分退款后，订单支付状态进入部分退款。
- 全额退款后，对应订单商品进入已退款。
- 退款完成后必须反向扣除相关方佣金。

### 分佣扣回与押金规则

- 售后中的订单不影响已生成分账。
- 售后退款完成后，按退款商品和退款金额扣回自提点、团长、供应商等相关方佣金。
- 押金是各方余额的一部分，但押金不可提现。
- 只有售后扣回场景允许各方佣金余额变为负数。
- 提现不能让佣金余额变为负数。
- 押金不足或佣金倒扣后余额为负时，进入资金异常，后续收益优先冲抵负数。
- 除非解除合同并手动释放押金，否则押金不允许提走。
- 佣金扣回必须保留原分佣记录和扣回记录，不能覆盖原记录。

### 退货入库规则

- 退货退款需要仓库或自提点确认退货后，再进入退款流程。
- 退货商品按实际情况进入可售库存、异常库存或报损。
- 退货入库必须写库存流水。
- 生鲜、临期、破损、过期商品默认不能重新入可售库存。

### 异常联动

- 缺货、破损、错货、少件、退款失败、押金不足、责任方争议进入异常中心。
- 售后单必须关联订单、订单商品、团期、SKU、自提点、供应商、仓库、配送任务。
- 异常中心只处理需要运营介入的问题，普通售后审核仍在售后中心处理。

### 后台能力

- 售后中心支持按团期、城市、自提点、供应商、仓库、售后类型、售后状态、责任方筛选。
- 支持查看订单商品快照、支付流水、退款记录、配送记录、分佣扣回记录。
- 审核、退款、驳回、补货、责任方认定必须做按钮权限。
- 售后导出涉及用户、手机号、金额时必须记录导出日志。

### 核心指标口径

- GMV：已支付订单商品实付金额，不含未支付订单，不含已取消未支付订单。
- 支付金额：实际支付成功金额，以支付流水为准。
- 退款金额：退款成功金额，以退款流水为准。
- 净销售额：支付金额 - 退款成功金额。
- 订单数：按订单主表统计。
- 商品销量：按订单商品明细统计。
- 客单价：支付金额 / 支付订单数。
- 缺货率：缺货订单商品数 / 应履约订单商品数。
- 履约率：已自提订单商品数 / 应自提订单商品数。
- 售后率：发起售后的订单商品数 / 已收货订单商品数。
- 退款率：退款成功金额 / 支付金额。
- 自提点履约率：该自提点已自提订单商品数 / 该自提点应自提订单商品数。
- 供应商缺货率：该供应商缺货订单商品数 / 该供应商应供订单商品数。

### 时间口径

- 经营报表默认按支付时间统计。
- 团期报表按团期截团时间和配送日都可筛选，默认按截团时间。
- 履约报表按履约节点时间统计，例如入库时间、出库时间、司机签到时间、用户自提时间。
- 售后报表按售后申请时间统计，退款金额另支持按退款成功时间。
- 财务报表按资金流水时间统计。
- 报表默认展示今日、近 7 日、近 30 日，可自定义时间范围。

### 权限与导出

- 平台管理员可看全平台报表。
- 城市运营只看当前城市及下级城市。
- 供应商只能看自身供应商相关报表。
- 自提点/团长只能看自身自提点报表。
- 财务可看资金类报表，但不默认开放商品运营编辑权限。
- 所有涉及订单、手机号、用户、财务、佣金、供应商结算的数据导出必须记录导出日志。
- 报表导出必须记录导出人、时间、报表类型、筛选条件、数据范围、导出原因。

### 边界规则

- 报表中心只展示和导出，不直接处理退款、提现、分账、缺货、售后。
- 对账差异去对账中心处理。
- 业务异常去异常中心处理。
- 报表数字允许有 T+1 汇总报表，实时经营指标以运营驾驶舱为准。
- 报表页面必须展示数据更新时间，避免运营误以为都是实时数据。

### 退款规则

- 售后审核仍在售后中心处理，财务风控不做普通售后审核。
- 高金额退款、批量退款第一版不做财务复核，沿用 v10。
- 退款必须关联原支付流水、订单商品、售后单或取消单。
- 退款失败进入异常资金和异常中心，可由有权限人员重试。
- 已支付未发货取消订单后，进入退款链路，退款流水必须可追踪。
- 财务不能直接改退款金额，退款金额以售后单、取消单计算结果为准。

### 提现规则

- 提现必须财务审核，不能自动通过。
- 提现对象包括供应商、自提点/团长等有佣金余额的角色。
- 可提现金额 = 可用余额 - 冻结金额 - 押金 - 风险冻结金额。
- 提现不能让佣金余额变成负数。
- 押金不能提现，除非解除合同并手动释放押金。
- 提现审核通过后进入打款流程，驳回后释放冻结金额。
- 提现失败进入异常资金，可重新打款或驳回退回余额。

### 分账与佣金规则

- 分佣按订单商品维度计算。
- 订单商品已自提后，可以先生成该订单商品的佣金明细。
- 订单主表全部商品已自提后，才生成最终分账。
- 未完成订单不生成最终分账。
- 售后退款完成后，按退款商品和退款金额扣回相关方佣金。
- 售后扣回允许佣金余额为负数，后续收益优先冲抵。
- 佣金重算必须保留原记录和重算记录，不能覆盖历史流水。

### 押金规则

- 押金是余额的一部分，但不可提现。
- 押金用于覆盖售后扣回、佣金为负、履约责任等资金风险。
- 押金释放必须手动操作，并填写原因。
- 解除合同后才允许释放押金并提现。
- 押金不足或余额为负时，进入异常资金。

### 对账规则

- 支付、退款、提现、分账都必须有对账记录。
- 对账差异不能只显示在报表里，必须进入异常资金或异常中心。
- 对账差异处理必须记录原因、处理人、处理时间、关联流水。
- 财务对账只处理资金差异，不直接改订单、售后、采购、出库状态。

### 权限与日志

- 财务可处理提现、对账、异常资金、分账失败、退款失败。
- 普通运营不能处理提现审核、押金释放、分账重试、对账差异。
- 退款重试、提现审核、押金释放、佣金重算、分账重试必须有按钮权限。
- 所有资金动作必须记录操作日志。
- 财务数据导出必须记录导出原因、筛选条件、数据范围和操作人。

### 边界规则

- 财务风控不替代售后中心，售后是否通过由售后中心决定。
- 财务风控不替代异常中心，异常中心负责业务闭环，财务风控负责资金处理。
- 报表中心只看结果，资金差异必须回到对账中心或异常资金处理。
- 第一版先保证资金可追踪、可审核、可重试、可对账，不做复杂自动风控模型。

### 角色规则

- 供应商可进入运营后台 PC 端，但只能看和操作自身供应商资料、商品协同、采购单、送货单、售后责任、结算数据。

- 涉及手机号、地址、结算账户、资金数据的导出必须记录导出原因和筛选条件。

- 自提点、仓库等端的独立登录规则后续按对应端单独拍板。
- 第一版重点是账号清晰、权限可控、供应商可登录但隔离、仓库角色不混入运营后台、操作留痕。

### 功能范围

| 页面 | 第一版规则 |
|------|------------|
| 商品中心 | 商品基础资料、商品状态、图片、描述、规格、供应商、类目 |
| SKU 管理 | SKU 编码、销售规格、库存规格、转换系数、条码、单位、保质期、生产日期字段 |
| 类目中心 | 前台展示类目、后台统计类目 |
| 标签管理 | 榜单、推荐、新品、热销等展示标签 |

### 商品与团期关系

- 商品是基础资料，团期是真正可售单元，沿用 v6。
- 一个商品可以进入多个团期。
- 同一个商品不能在同一时间段售卖。
- 商品资料停用后，不允许进入新团期。
- 商品资料停用不影响已发布团期和已支付订单履约。
- 已进入团期的商品，商品关键资料修改必须保留快照，不能回改历史订单商品信息。

### 商品字段规则

- 商品名称、主图、规格、单位、供应商、统计类目、前台展示类目、保质期相关信息必须完整。
- 一个商品必须绑定统计类目。
- 一个商品必须绑定前台展示类目。
- 生鲜、食品类商品必须维护生产日期、保质期或保质期规则。
- 商品图片、详情描述用于前台展示，但实际售卖价格、库存、配送日期由团期决定。
- 商品可以配置默认供应商，但采购单最终以团期截团生成结果为准。
- 商品可以配置建议零售价或参考价，但团期价格才是实际售卖价格。

### SKU 与库存规格规则

- SKU 是库存、采购、订单商品、售后、分佣、报表统计的最小商品单位。
- SKU 编码必须唯一。
- 同一商品可有多个 SKU。
- 商品销售规格必须绑定一个库存规格。
- 销售规格是前台售卖、订单商品、团期发布使用的规格口径。
- 库存规格是仓库入库、库存管理、库存流水、出库扣减后续使用的规格口径。
- 销售规格和库存规格之间必须有转换系数，用于数量换算。
- 可以多个销售规格绑定同一个库存规格，但转换系数可以不同。
- 同一个库存规格只能被同一个 SPU 下的 SKU 绑定，不允许跨 SPU 共用库存规格。
- 如果运营创建商品规格时没有手动绑定库存规格，系统默认生成一个库存规格并自动绑定，转换系数默认为 1:1。
- 默认生成的库存规格后续可用于仓库入库、库存流水、出库等操作。
- SKU 停用后不能进入新团期，但历史订单、采购、售后、库存流水仍可追溯。
- SKU 关键规格修改后，历史订单商品快照不回改。
- SKU 合并、拆分第一版不做，避免库存和财务口径混乱。
- 销售规格绑定库存规格后，历史订单商品快照不回改。

### 类目规则

- 前台展示类目用于小程序端商品分类展示。
- 后台统计类目用于售后规则、报表统计、经营分析。
- 售后规则按统计类目配置，沿用 v6 和 v12。
- 一个商品必须绑定统计类目。
- 一个商品必须绑定前台展示类目。
- 前台类目调整只影响展示，不回改历史订单。
- 统计类目变更必须二次确认，因为会影响后续售后规则和报表统计。

### 商品状态

| 状态 | 规则 |
|------|------|
| 待审核 | 商品所有必填数据填写完整后，才允许提交审核；审核通过后变为已启用 |
| 已启用 | 可进入团期发布 |
| 已停用 | 不可进入新团期，历史数据保留 |
| 已删除 | 第一版只做逻辑删除，不做物理删除 |

### 审核规则

- 商品新增后进入待审核。
- 待审核商品必须完成所有必填数据，才允许审核通过变为已启用。
- 必填数据不完整时，不允许审核通过。
- 审核驳回后仍保持待审核状态，需补齐或修改后再次审核。
- 已启用商品的关键资料变更，需要重新审核或二次确认。
- 已启用商品变更统计类目、前台展示类目、供应商、规格、保质期规则时，必须记录原因和修改前后内容。

### 发布前校验关系

- 团期发布时校验商品状态必须为已启用。
- 团期发布时校验 SKU、销售规格、库存规格、转换系数、供应商、统计类目、前台展示类目完整。
- 团期发布时必须校验 SKU 已绑定库存规格。
- 团期发布时必须校验销售规格与库存规格转换系数有效。
- 如果运营未绑定库存规格，系统必须先自动生成默认库存规格并绑定，再允许进入团期发布。
- 同一 SPU 下多个 SKU 可以绑定同一个库存规格。
- 不同 SPU 的 SKU 不允许绑定同一个库存规格。
- 团期发布时不校验商品级售后规则，因为售后规则按统计类目默认配置。
- 团期发布时不配置自提点黑白名单，沿用 v6。

### 仓库边界

- 商品中心只维护销售规格、库存规格和转换系数。
- 仓库后台后续按库存规格做入库、库存、出库、库存流水。
- 商品中心不处理实际库存数量调整。
- 转换系数变更必须二次确认，并记录修改前后值、操作人和原因。
- 已有库存流水或已生成订单的规格，不建议直接修改转换系数；确需修改时必须保留历史转换快照。

### 权限与日志

- 商品新增、编辑、停用、启用、删除必须有按钮权限。
- 商品停用、统计类目变更、前台展示类目变更、供应商变更、转换系数变更必须二次确认。
- 商品导出需要导出权限和导出日志。
- 商品关键资料修改必须记录修改前后内容、操作人、原因和时间。
- 供应商只能维护或查看自身商品协同范围内的数据，不能修改平台统一规则。

### 边界规则

- 商品中心不处理团期价格、团期库存、截团、采购生成、订单履约。
- 商品中心不处理库存调整，库存调整归库存/仓库相关流程。
- 商品中心不处理售后审核，只提供统计类目和商品快照。
- 前台、订单、团期使用销售规格；仓库、入库、库存流水、出库使用库存规格；两边通过转换系数换算。
- 第一版重点是商品资料准确、SKU 口径清晰、类目规则稳定、历史快照可追溯。

- 销售开始时间。
- 销售结束日期/结束时间。
- 截团时间不在团期里手动填写，由系统按“大区截团时间 + 销售结束日期”生成实际截团时刻。
- 采购仓。
- 库存售卖模式：虚拟库存或仓库库存。
- 商品/SKU。
- 团期价格。
- 限购规则。
- 起售数量。
- 商品销售规格、库存规格、转换系数必须完整。
- 配送日期非必填，不填默认截团后配送；如果指定，配送日期必须大于实际截团日期。

### 商品进入团期规则

- 只有已启用商品允许进入团期。
- 商品必须绑定统计类目和前台展示类目。
- SKU 必须绑定库存规格。
- 销售规格和库存规格必须有有效转换系数。
- 商品/SKU 停用后，不允许进入新团期。
- 已进入团期的商品资料、SKU、类目、库存规格、转换系数必须做快照。
- 同一个商品不能在同一时间段重复售卖。

### 截团时间规则

- 大区配置里的截团时间只保存时分秒，例如 22:00:00。
- 团期不手动填写截团时间。
- 团期允许销售多天。
- 团期实际截团时刻 = 团期销售结束日期 + 大区配置截团时间。
- 示例：销售结束日期是 2026-06-23，大区截团时间是 22:00:00，系统实际截团时刻为 2026-06-23 22:00:00。
- 大区未配置截团时间时，不允许发布团期。
- 团期发布时保存大区截团时间快照，快照只保存时分秒配置，同时业务计算时记录实际截团时刻。
- 后续修改大区截团时间，不影响已发布团期。
- 不校验“销售结束时间必须小于或等于截团时间”。
- 只校验销售开始时间必须小于销售结束时间。
- 如果指定配送日期，配送日期必须大于实际截团日期。
- 第一版一个团期只允许覆盖同一大区下的城市。
- 如果要多个大区售卖同一批商品，需要拆成多个团期。

### 库存模式规则

| 模式 | 规则 |
|------|------|
| 虚拟库存 | 用于预售集采，销售数量形成采购建议量，截团后按配送日生成采购单 |
| 仓库库存 | 用于现货售卖，按采购仓可售库存控制，截团后生成出库单和司机配送数据 |

- 团期发布前必须明确库存模式。
- 团期发布后不允许切换库存模式。
- 仓库库存模式必须校验采购仓可售库存。
- 虚拟库存模式不直接占用仓库库存，但必须形成采购需求。
- 库存调整按 v4、v9、v20 规则执行，必须留流水或操作日志。

### 时间规则

- 销售开始时间必须小于销售结束时间。
- 团期允许销售多天。
- 不再校验销售结束时间必须小于或等于截团时间。
- 如果指定配送日期，配送日期必须大于实际截团日期。
- 同一商品多个团期销售时间段不允许重叠。
- 团期到实际截团时刻后，由系统任务触发汇总订单和后续采购/出库链路。
- 团期过期后不可继续下单。

### 发布前校验

- 校验商品状态为已启用。
- 校验商品必填资料完整。
- 校验统计类目和前台展示类目完整。
- 校验 SKU、销售规格、库存规格、转换系数完整。
- 校验销售开始时间、销售结束时间、实际截团时刻、配送日期。
- 校验团期价格、限购、起售数量。
- 校验库存模式和采购仓。
- 校验大区是否已配置截团时间。
- 校验同一商品是否存在同时间段售卖的其他团期。
- 不校验商品级售后规则，系统使用统计类目配置的默认售后规则。
- 不校验自提点范围，团期第一版不做自提点黑白名单。

### 发布后修改规则

| 团期状态 | 允许修改 |
|----------|----------|
| 待发布 | 可修改全部团期配置 |
| 已发布且未开始销售 | 可修改价格、库存、销售时间、配送日期、商品池 |
| 已开始销售未截团 | 只能修改商品价格、库存、生产日期、保质期数据 |
| 已截团 | 不允许修改团期核心配置，只能查看和处理异常 |
| 已下架 | 不允许继续售卖，已支付订单继续履约 |
| 已过期 | 只读查看，不允许重新发布 |

### 团期状态

- 待发布。
- 已发布。
- 已下架。
- 已过期。

说明：已截团是业务节点，不作为团期主状态；用于限制修改和触发采购/出库/配送链路。

### 下架与过期规则

- 团期下架后，小程序端不可继续购买。
- 团期下架不影响已支付订单履约。
- 团期下架必须二次确认，并记录原因。
- 团期到期后系统自动变为已过期。
- 已过期团期只读展示，可用于报表和追溯。

### 截团后链路

- 系统汇总团期订单。
- 虚拟库存模式：截团后根据配送日生成采购单。
- 仓库库存模式：按采购仓可售库存生成出库单和司机配送数据。
- 所有采购单生成都需要扣减同一配送日同一 SKU 的采购数量，避免重复采购。
- 缺货时进入异常中心，由运营发起补货，采购生成采购单。
- 司机配送数据在截团时生成，出库完成后司机才能打卡发车。
- 配送可能后续由仓库后台按线路处理，运营后台只关心采购仓和结果异常。

### 权限与日志

- 团期发布、修改、下架、库存调整、价格调整必须有按钮权限。
- 团期发布、下架、库存模式选择、库存调整必须记录操作日志。
- 团期下架、已发布团期关键修改必须二次确认。
- 团期导出必须有导出权限和导出日志。
- 城市运营只能管理自己城市范围内团期。
- 供应商只能查看或协同自身相关团期数据，不能发布平台团期。

### 边界规则

- 团期管理不维护商品基础资料，商品资料归商品中心。
- 团期管理不处理实际仓库入库、出库、司机配送操作，仓库后台处理。
- 团期管理不处理售后审核，售后中心处理。
- 团期管理不处理财务分账和提现，财务风控处理。
- 第一版重点是发布可控、售卖时间不冲突、库存模式清晰、截团后链路可追溯。

### 订单状态规则

| 主状态 | 规则 |
|--------|------|
| 未支付 | 用户已下单未支付，15 分钟超时系统自动关闭 |
| 已付款/待发货 | 支付成功，未进入发货履约完成节点 |

### 订单商品状态

- 订单商品是履约、售后、退款、分佣、报表统计的关键维度。
- 同一订单下不同商品可以有不同履约状态。
- 订单商品状态只表示履约进度。
- 订单商品状态包括：待履约、待出库、配送中、待自提、已自提。
- 售后中、已退款、已关闭不放在订单商品履约状态里。
- 售后中、已退款、已关闭在售后单/售后订单里展示。
- 订单详情可以展示该订单商品是否有关联售后，但不改变订单商品履约状态。
- 订单商品已自提后，用户端可以针对该订单商品发起售后，沿用 v10。
- 订单主表已完成，必须以订单下所有商品都已自提为准。

### 取消规则

- 未支付订单只能允许用户取消。
- 未支付订单 15 分钟超时关闭属于系统自动关闭，不属于运营后台手动取消。
- 已支付未发货订单，运营后台允许通过按钮权限取消。
- 已支付未发货订单后台取消后，进入退款取消链路，必须生成取消记录和退款流水。
- 已发货后禁止取消，只能走售后。
- 运营取消已支付未发货订单必须填写原因、二次确认、记录操作日志。

### 履约节点规则

- 支付成功后，订单进入待履约。
- 到实际截团时刻后，系统汇总团期订单。
- 虚拟库存模式：截团后根据配送日生成采购单。
- 仓库库存模式：按采购仓可售库存生成出库单和司机配送数据。
- 司机配送数据在截团时生成。
- 出库完成后司机才能打卡发车。
- 司机送达自提点后，司机签到即默认自提点签收。
- 司机签到后，对应订单商品进入待自提。
- 用户自提后，订单商品进入已自提。

### 完成规则

- 订单商品已自提后，可先生成该订单商品的佣金明细。
- 订单下所有订单商品履约状态都为已自提后，订单主表进入已完成。
- 售后中不影响订单主表完成状态。
- 已完成订单商品发生退款后，不回退订单履约状态。
- 未完成订单不生成最终分账。
- 售后退款完成后，按退款商品和退款金额扣回佣金，沿用 v10 和 v16。

### 快照规则

- 订单必须保存下单时的商品、SKU、销售规格、库存规格、转换系数、价格、类目、供应商、团期、售后规则、佣金规则快照。
- 后续商品、类目、规则、转换系数变化，不回改历史订单。
- 订单详情必须能看到使用的团期、规则版本和关键快照。

### 权限与日志

- 订单查看、导出、取消、查看完整手机号必须分开授权。
- 已支付未发货取消必须有独立按钮权限。
- 查看完整手机号、导出订单、后台取消订单必须记录日志。
- 城市运营只看本城市及下级订单。
- 供应商只能看自身供应商相关订单商品数据，不能看平台全量订单。
- 自提点/团长不进入运营后台 PC 端，相关订单由自提点小程序查看。

### 边界规则

- 订单中心不处理仓库入库、出库、司机配送实操。
- 订单中心不审核售后，售后中心处理。
- 订单中心不处理提现、分账、押金释放，财务风控处理。
- 订单中心不直接改库存，只展示库存和履约结果。
- 第一版重点是订单状态清晰、订单商品维度可追踪、取消规则受控、履约节点可追溯。

| 页面 | 第一版规则 |

### 售后发起规则

- 售后按订单商品维度发起。
- 订单商品已自提后，用户端可以针对该订单商品发起售后。
- 已退款或已关闭售后的订单商品，允许再次发起售后。
- 售后进行中时，同一订单商品不允许同时再次发起售后。
- 超出售后期后，不允许再发起售后。
- 售后期按统计类目配置，沿用 v6、v12、v20。

### 售后类型

| 售后类型 | 第一版规则 |
|----------|------------|
| 仅退款 | 缺货、少件、破损、质量问题、过期、无法履约等场景 |
| 退货退款 | 需要退回商品后再退款 |
| 补货/补发 | 可重新履约的缺货、错货、少件场景 |

- 换货第一版不做独立流程，平台按补货单处理。
- 平台补偿第一版暂不做。

### 售后状态

| 状态 | 规则 |
|------|------|
| 待审核 | 用户提交售后后进入待审核 |
| 已驳回 | 审核驳回，需填写原因 |
| 退款中 | 审核通过并进入退款链路 |
| 退款成功 | 退款流水成功 |
| 退款失败 | 退款流水失败，可重试并进入异常中心 |
| 待退货 | 退货退款审核通过后，等待退货 |
| 已退货 | 已确认退货，允许进入退款 |
| 补货处理中 | 已生成补货单，等待履约 |
| 已补货 | 补货/补发完成 |
| 已关闭 | 用户撤销、超时关闭、人工关闭或流程结束关闭 |

### 审核规则

- 售后必须由运营/客服审核。
- 审核通过、驳回都必须填写原因。
- 退款金额不能超过该订单商品可退金额。
- 支持部分退款。
- 高金额退款或批量退款第一版不要求财务复核，沿用 v10。
- 涉及责任方争议、退款失败、补货异常，进入异常中心。

### 退款规则

- 退款必须关联原支付流水、订单商品、售后单。
- 退款失败进入异常中心和财务风控，可重试。
- 部分退款后，售后单记录部分退款金额。
- 全额退款后，对应售后单进入退款成功。
- 退款完成后必须反向扣除相关方佣金。
- 售后退款不回退订单商品履约状态，订单商品仍保持已自提。

### 退货退款规则

- 退货退款必须先审核通过，再确认退货，再发起退款。
- 退货商品按实际情况进入可售库存、异常库存或报损。
- 生鲜、临期、破损、过期商品默认不能重新进入可售库存。
- 退货入库和库存流水归仓库后台处理，售后中心只看结果。

### 补货/补发规则

- 补货/补发由售后审核通过后生成补货单。
- 补货单必须关联原订单商品、售后单、SKU、团期、自提点。
- 补货履约由仓库后台处理，售后中心只看补货状态。
- 补货失败或缺货进入异常中心。
- 补货完成后，售后单进入已补货或关闭。

### 责任方规则

- 售后可标记责任方：供应商、仓库、司机、自提点、平台、用户。
- 责任方用于报表统计、供应商责任、佣金扣回和异常分析。
- 责任方变更必须记录原因和操作日志。
- 责任方争议进入异常中心。

### 佣金扣回规则

- 售后中的订单不影响已生成分账。
- 售后退款完成后，按退款商品和退款金额扣回相关方佣金。
- 售后扣回允许佣金余额为负数，后续收益优先冲抵。
- 押金不足或余额为负时，进入异常资金。
- 提现不能让佣金余额变负，沿用 v16。
- 佣金扣回必须保留原分佣记录和扣回记录，不能覆盖原记录。

### 权限与日志

### 押金释放规则

- 押金默认不能提现。
- 解除合同后，才允许手动释放押金。
- 押金释放必须由财务或平台管理员发起。
- 押金释放必须填写释放原因。
- 押金释放必须经过审批。
- 押金释放后，该部分金额转为可提现余额。
- 押金释放后，允许按提现流程提走。
- 押金释放必须记录押金流水、资金流水、操作日志。

### 提现状态

- 待审核。
- 已驳回。
- 待打款。
- 打款中。
- 打款成功。
- 打款失败。
- 已取消。
- 已关闭。

### 边界规则

- 提现中心不直接处理佣金计算。
- 提现中心不直接处理售后扣回。
- 提现中心不替代资金流水，提现结果必须形成资金流水。
- 押金释放不等于提现，释放后只是转为可提现余额。
- 第一版重点是控制可提现余额、押金余额、负佣金冲抵、提现审核和打款失败重试。

### 分佣对象

| 对象 | 第一版规则 |
|------|------------|
| 自提点 | 按订单商品、自提点佣金规则计算 |
| 团长 | 按订单商品、团长绑定关系和佣金规则计算 |
| 供应商 | 按订单商品、供应商结算规则计算 |
| 手续费 | 按订单支付通道、平台配置或实际支付成本记录 |
| 平台 | 按实付金额扣除自提点佣金、团长佣金、手续费、供应商佣金后的剩余金额计算 |
| 城市运营 | 第一版不做城市运营分佣 |
| 用户 | 第一版不做用户推广佣金 |

### 平台分佣公式

- 平台分佣按订单商品维度计算。
- 公式：平台分佣 = 实付金额 - 自提点佣金 - 团长佣金 - 手续费 - 供应商佣金。
- 平台分佣不能再用固定比例直接计算。
- 手续费作为独立分佣对象和资金成本项记录。
- 手续费必须能追溯到支付流水或手续费配置快照。
- 分佣明细必须展示实付金额、自提点佣金、团长佣金、手续费、供应商佣金、平台分佣。
- 如果售后退款，手续费是否退回按支付通道实际结果记录，平台分佣按退款后的结果重新扣回或调整。

### 分佣触发规则

- 订单商品已自提后，可先计算该订单商品佣金。
- 订单商品维度允许先形成佣金流水。
- 订单主表未完成前，不生成最终分账。
- 订单下所有商品都已自提后，订单主表进入已完成状态。
- 订单主表已完成后，才允许生成最终分账。
- 有售后中的订单不影响已完成订单的分账流程。
- 售后退款成功后，再按退款结果扣回对应佣金。

### 佣金计算规则

- 佣金按订单商品维度计算。
- 佣金规则按订单商品创建时的规则快照执行。
- 自提点佣金按订单商品所属自提点计算。
- 团长佣金按用户下单时绑定的团长关系计算。
- 一个团长只能绑定一个自提点，一个用户只能绑定一个团长，沿用 v17。
- 供应商结算按供应商、商品、采购/结算规则计算。
- 手续费按支付通道结果、手续费配置或实际支付成本记录。
- 平台分佣按实付金额扣除其他分佣对象后剩余金额计算。
- 佣金金额必须记录计算口径、比例、基数、快照版本。
- 佣金计算结果必须可追溯到订单、订单商品、团期、SKU、规则版本。

### 售后扣回规则

- 售后退款成功后，按退款订单商品扣回各方佣金。
- 仅售后场景允许各方佣金余额为负数。
- 提现不能让佣金余额变负数。
- 押金不能直接提现，押金规则沿用 v32。
- 佣金扣回必须形成佣金流水和资金流水。
- 手续费是否退回按支付通道实际结果记录。
- 售后关闭或售后驳回，不扣回佣金。
- 同一售后单不能重复扣回同一笔佣金。

### 分账规则

- 最终分账只在订单主表已完成后生成。
- 分账按订单商品汇总各方应得金额。
- 分账前必须校验资金流水、佣金流水、手续费记录、售后扣回结果。
- 分账失败进入异常中心或系统任务重试。
- 分账失败可由财务按权限重试。
- 分账重试必须记录操作日志和资金流水状态。
- 分账成功后，生成分账流水。
- 分账流水不允许删除或直接改金额。

### 边界规则

- 分佣中心不处理提现审核，提现按 v32。
- 分佣中心不处理售后审核，售后按 v23。
- 分佣中心不处理采购入库和供应商送货。
- 手续费只作为分佣和资金成本口径记录，不替代支付通道对账。
- 报表中心只引用分佣、分账、手续费、资金流水统计结果。
- 第一版重点是订单商品维度算佣金、手续费单独记录、平台按剩余金额计算、订单完成后最终分账、售后退款后可扣回，先不做复杂多级分销。

| 执行原则 | 供应商结算以实际入库结果和确认采购价为准；采购协同只展示结算状态，结算入账、扣回、提现归财务风控处理 |

### 结算对象

| 对象 | 第一版规则 |
|------|------------|
| 供应商 | 按供应商主体结算 |
| 采购单 | 作为结算来源之一 |
| 送货单 | 作为入库和差异追踪来源 |
| 入库结果 | 作为最终可结算数量来源 |
| 售后责任 | 作为后续扣回或责任追踪来源 |
| 采购异常 | 作为扣减、补货、拒收、关闭异常依据 |

### 结算金额规则

- 供应商结算金额 = 实际入库可结算数量 × 确认采购价。
- 实际入库可结算数量以仓库入库结果为准。
- 确认采购价以采购审核通过后的采购价为准。
- 采购价调整必须有独立权限、审批或审核记录、操作日志。
- 多送、少送、破损、拒收、过期、规格不符不直接结算，按入库结果和异常处理结果计算。
- 供应商自建采购单审核通过后，才允许进入后续结算链路。
- 供应商未确认或采购未审核通过的采购单，不形成结算金额。

### 入账规则

- 仓库完成入库后，系统生成供应商待结算金额。
- 采购异常未关闭时，对应异常数量暂不进入可结算余额。
- 采购异常关闭后，按处理结果生成结算、扣减或补货记录。
- 供应商结算余额进入供应商账户余额。
- 供应商可提现余额按 v32 提现规则执行。
- 供应商结算入账必须生成资金流水或结算流水。
- 供应商结算结果必须可追溯到采购单、送货单、入库单、SKU、库存规格、转换系数。

### 售后责任扣回

- 售后判定为供应商责任时，可从供应商结算余额或后续结算中扣回。
- 供应商责任扣回必须关联售后单、订单商品、SKU、责任原因。
- 扣回必须生成佣金/结算扣回流水和资金流水。
- 售后关闭或非供应商责任，不扣回供应商结算。
- 同一售后单不能重复扣回同一笔供应商结算。
- 扣回导致供应商余额为负时，后续结算优先冲抵。

### 结算状态

- 待入库。
- 待结算。
- 部分结算。
- 已结算。
- 暂缓结算。
- 已扣回。
- 异常。
- 已关闭。

说明：结算状态不替代采购单、送货单、入库单、售后单状态。

### 权限与日志

- 供应商可查看自身结算数据。
- 供应商不能审核、调整、释放、冲正自己的结算金额。
- 采购人员可查看采购相关结算状态，但不能处理提现和押金释放。
- 财务可处理结算入账、扣回、提现、对账差异。
- 结算调整必须有审批、原因、操作日志。
- 结算数据导出不脱敏，沿用 v29。
- 结算流水不允许删除或直接改金额。

### 边界规则

- 供应商结算不替代采购审核。
- 供应商结算不替代仓库入库。
- 供应商结算不替代售后责任判定。
- 供应商结算不替代提现审核。
- 第一版重点是实际入库数量、确认采购价、异常扣减、售后责任扣回、供应商可提现余额口径统一。

- 停用或拉黑不影响历史订单、售后、结算继续闭环。
- 停用、启用、拉黑、解除拉黑必须有原因和操作日志。
- 供应商存在严重缺货、资质造假、结算异常、售后责任拒不处理时，可进入黑名单。

- 供应商不能审核自己的资质、合同、押金、黑名单状态。
- 运营可维护供应商主体和状态。
- 财务可维护结算账户、押金、账期、结算相关字段。

- 供应商合同不替代采购审核。

- 商品待审核必须所有必填数据完整后才允许审核通过。
- 商品必须绑定统计类目和前台展示类目。
- 商品规格必须绑定库存规格。
- 团期状态只保留：待发布、已发布、已下架、已过期。
- 团期发布后不允许切换库存模式。
- 已开始销售未截团的团期，只允许修改已拍板允许修改的字段。

### 订单与履约状态

- 订单商品状态只保留：待履约、待出库、配送中、待自提、已自提。
- 售后中、已退款、已关闭不放在订单商品状态里，只在售后订单中展示。
- 订单主表只有订单下所有商品都已自提后才进入已完成。
- 未支付订单只允许用户取消。
- 已支付未发货订单运营后台允许按权限取消。
- 已发货后不允许取消，只能走售后。

### 售后状态和按钮

- 售后按订单商品维度展示。
- 已收货订单商品允许发起售后。
- 退款待审核等处理中状态不允许重复发起。
- 已退款或已关闭售后的商品允许再次发起。
- 超出售后期不允许发起。
- 售后审核、退款、退款重试必须有明确按钮和状态。
- 换货不做独立流程，平台按补货单处理。
- 平台补偿第一版不做。

### 采购、出库、配送按钮

- 供应商可创建采购单，但必须采购审核。
- 采购审核通过后才进入正式采购流程。
- 截团后生成采购、出库、司机配送数据。
- 出库完成后司机才能打卡发车。
- 司机送达自提点后签到，默认自提点签收。
- 缺货、破损等异常不在司机签收环节复杂处理，由用户端发起售后。
- 缺货进入异常中心，由运营发起补货。

### 财务、分佣、提现按钮

- 分账、结算、提现、押金释放属于高风险操作。
- 提现审核、打款失败重试、押金释放必须有操作按钮、确认提示和日志。
- 提现不能让佣金余额变负。
- 押金不能提现，解除合同并手动释放押金后才允许提走。
- 售后退款后允许佣金扣回为负。
- 平台分佣按已拍板公式展示和计算。

### 通用按钮规则

- 新增、编辑、审核、启用、停用、发布、下架、取消、退款、导出、导入、重试等按钮必须受角色和数据范围控制。
- 无权限按钮可以隐藏，或展示后点击提示无权限，但不能绕过权限。
- 已拍板不允许操作的场景，按钮必须隐藏、置灰或给出明确原因。
- 高风险按钮必须二次确认或审批。
- 批量操作只能覆盖低风险动作。
- 第一版不做的功能不能出现灰色占位按钮。

### 定版阻断项

以下问题出现，第一版不能定版：

- UI 状态和已拍板状态不一致。
- 订单商品状态混入售后状态。
- 团期出现未拍板状态。
- 商品还保留草稿状态。
- 不做项按钮或入口仍然存在。
- 高风险操作没有确认、审批或日志。
- 按钮可以绕过角色、数据范围或业务状态机。
- 敏感字段被错误脱敏，或字段级权限控制被误做成第一版要求。

### 可后置项

- 字段排序优化。
- 表格列宽优化。
- 按钮位置微调。
- 提示文案精修。

- 非核心批量操作补充。

- 如果能力只在说明文案里出现，但没有入口、Tab、详情或可操作位置，仍按缺失处理。
- 如果能力入口存在但无法追溯到业务对象，按追溯差异处理。
- 如果能力入口存在但受错误权限、错误状态或错误按钮控制，按权限/按钮差异处理。
- 如果能力入口存在但属于第一版明确不做项，仍按必须隐藏处理。

### 菜单验收调整

- 一级菜单仍固定为 8 个。
- 二级菜单数量不作为硬性验收标准。
- 详情类、记录类、配置类、统计类能力优先放在对应业务主页面中承接。
- 不允许为了满足页面清单而新增空页面、灰色占位页或敬请期待页。
- 同一能力可以从多个详情页跳转进入，但必须有一个明确主承接页面。

### 差异清单影响

- 后续差异清单不再把“未独立出现在二级菜单”直接判为 P1。
- 后续差异清单改为判断能力是否存在、是否可进入、是否可操作、是否可追溯。
- 若页面内已承接但命名不清，按 P2 文案或导航优化处理。
- 若页面内未承接且属于第一版必做能力，按 P1 必须补齐处理。
