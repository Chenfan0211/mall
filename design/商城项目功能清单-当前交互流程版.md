# 商城项目功能清单（当前交互流程版）

编制日期：2026-06-15

本文档根据当前《商城项目交互原型图.html》的页面、菜单、按钮、移动端角色切换和 WMS 单据流转整理，作为后续需求确认、接口拆分、数据库设计和开发排期的功能范围依据。

## 1. 系统范围

本项目是社区团购/商城 + 进销存配一体化系统，覆盖商品建档、商品发布、用户下单、支付、积分、采购、入库、仓储作业、配送、自提点交接、售后退货、分账佣金、提现、司机费用、经营报表和系统自动任务。

| 端口 | 使用角色 | 核心职责 |
| --- | --- | --- |
| 运营后台 | 平台运营、城市运营、采购、财务、管理员 | 运营驾驶舱、商品与团期、交易履约、自提点/团长、供应商/采购、财务风控、用户与营销、报表与系统 |
| 用户小程序 / H5 | 普通用户 | 浏览商品、切换城市/自提点、领券、加购、下单、支付、查看订单、查看积分、申请售后、评价、分享 |
| 角色工作台小程序 | 自提点、供应商 | 自提点订单/收货/退货/佣金/休假；供应商送货单/商品销售/缺货/发票/提现 |
| 仓配移动端 H5 | 收货员、采购员、拣货员、复检装车员、司机、仓管 | 采购、收货、质检、上架、拣货、缺货、复检、装车、配送、退货、库存、盘点 |
| WMS 后台 | 仓库主管、仓配运营、物流调度 | 单据流转、采购、入库、库存、出库、配送、车辆、线路、退货、司机费用、日志 |
| 系统任务 | 定时任务、MQ、缓存、搜索服务 | 商品 ES 同步、Redis 缓存刷新、自动取消订单、生成配送数据、分账重试、消息通知 |

## 2. 全链路业务流程

1. 运营后台建模：后台按 8 个一级菜单组织运营驾驶舱、商品与团期、交易履约、自提点/团长、供应商/采购、财务风控、用户与营销、报表与系统。
2. 商品建档：运营维护商品、品牌、供应商、采购员、展示类目、统计类目、商品规格、库存规格、换算系数、城市价格、长宽高、重量、榜单标签、联想词。
3. 团期管理：运营创建团期，设置销售城市、销售开始/结束时间、截团时间、商品池、采购仓/履约仓、库存售卖模式、价格、限购和起售数量；配送日期非必填，不配置时默认截团后配送；第一版不做团期自提点范围和商品级售后规则；发布前完成校验后刷新小程序商品列表；团期管理承接原商品发布能力。
4. 用户下单：用户在小程序浏览首页、分类、活动、榜单、商品详情，加入购物车，提交订单，选择优惠券、提货人和订单备注，完成支付；提交订单前强校验价格、库存、上下架状态、销售期、限购数和商品城市；从待支付订单继续支付时沿用订单快照，不重复校验商品上下架、库存、价格、销售期和自提点城市。
5. 交易履约：订单按交易状态、支付状态、履约状态三条线判断；支付成功后生成订单、支付流水、商品快照、商品级分佣预估和待到账积分记录；截团后联动采购、入库、出库、司机配送、司机签到默认签收和缺货异常。
6. 采购入库：采购单来源必须可追溯，来源包括系统生成、运营发起、采购员创建和供应商创建；供应商可为自身 SKU 创建采购单，但必须采购审核通过后才转正式采购单，审核前不占用采购数量、不生成送货单；仓库按采购单送货入库、补货采购入库、退货入库、盘点调整入库处理收货，记录差异、确认入库和写库存流水。
7. 库存出库：仓库库存模式截团后按团期、配送日、自提点生成出库单；虚拟库存模式采购入库后生成出库单；库存按 SKU、仓库、批次、生产日期、保质期管理，出库按 FIFO 扣减库存。
8. 司机配送：司机配送数据在截团时按配送日、线路、自提点、司机生成；出库未完成前司机可查看任务但不能打卡发车，出库完成后才允许打卡发车并进入配送中。
9. 自提点交付：司机送达自提点后签到即默认自提点签收，对应订单明细进入待自提；第一版不再单独做自提点强制确认签收，用户发现缺货、破损、错货后从用户端发起售后。
10. 售后退款：订单商品已收货后即可按订单商品维度发起售后；仅退款、退货退款、补货/补发由售后中心处理；同一订单商品售后进行中不允许并发发起，已退款或已关闭后允许再次发起。
11. 财务风控：订单主表已完成后生成最终分账，售后退款完成后按退款商品和退款金额扣回对应角色佣金；押金不可提现，只有售后扣回场景允许佣金余额为负，提现不能让佣金余额变负。
12. 积分运营：后台配置下单返积分、签到积分和过期规则，记录每个用户积分账户和每笔积分流水，过期任务按批次扣减。
13. 报表与系统：后台按订单、配送、佣金、积分、售后、统计类目、库存、司机费用等维度查看和导出报表，并维护权限、基础配置、系统任务和操作日志。

| P1 | 分账中心、佣金中心 | 保留，移动到财务风控 |

| 规则 | 详细说明 |
| --- | --- |
| 第一轮 UI | 只做 P0 页面和现有页面迁移 |

| 日志边界 | 系统任务和操作日志只作为辅助证据，不替代异常中心 |

### 3.1 运营驾驶舱

| 功能 | 详细说明 |
| --- | --- |
| 顶部核心指标 | 固定 6 个：今日 GMV、今日订单数、退款率、缺货率、自提点履约率、待处理异常数；顶部不继续堆叠更多数字。 |
| 趋势指标 | 展示 GMV、订单数、退款率、缺货率的 7 日趋势，点击趋势卡片进入订单、售后、缺货或报表明细。 |
| 待办任务 | 首页必须展示待办任务，按优先级展示待审核团期、待处理售后、待审核提现、待处理缺货、待确认采购、待处理异常。 |
| 异常预警 | 首页必须展示异常预警，覆盖库存不足、配送延误、退款失败、提现异常、分账失败、低评分评论；运营驾驶舱只展示高优先级和超时未处理异常，普通异常留在异常中心列表，每个异常都跳转到异常详情或对应处理页面。 |
| 城市排行 | 展示城市 GMV、订单数、履约率、退款率、缺货率，城市运营只能看本城市及下级数据。 |
| 自提点排行 | 展示高 GMV 自提点、低履约自提点、缺货高发自提点，点击进入自提点/团长或缺货处理页面。 |
| 快捷入口 | 固定入口：团期管理、订单中心、售后中心、异常中心、对账中心、自提点/团长管理。 |
| 默认筛选 | 默认看今日和当前用户数据权限范围；平台管理员看全局，城市运营看本城市及下级，供应商/自提点不进入运营驾驶舱。 |
| 刷新规则 | 首页指标支持手动刷新，关键异常 1 分钟内刷新或轮询。 |
| 空状态 | 无异常时展示“暂无待处理异常”，不要展示空表格。 |
| 红色预警 | 退款率、缺货率、履约率、提现异常、分账异常超过阈值时标红。 |

| 交互与风控规则 | 详细说明 |
| --- | --- |
| 可处理工作台 | 运营驾驶舱所有卡片都能点击跳转到对应模块，不能只做“好看但不能处理问题”的数据大屏。 |
| 优先级 | 财务异常、履约异常、缺货异常优先级高于普通增长指标。 |
| 运营视角 | 驾驶舱不展示供应商端、自提点端、仓库端专属明细，只展示运营视角汇总。 |
| 数据隔离 | 城市运营不能看全平台数据，数据范围必须跟当前用户权限一致。 |
| 异常字段 | 财务、履约、库存类异常必须展示责任模块、异常对象、发生时间、当前状态和处理入口。 |

### 3.1.1 异常中心闭环流程（2026-06-18 v11）

| 定位规则 | 拍板结论 |
| --- | --- |
| 执行原则 | 异常中心只处理需要运营介入的业务异常，不处理普通日志，不替代售后中心，不替代对账中心。 |
| 业务闭环 | 异常必须能从异常中心进入对应业务页面处理，并记录处理人、处理结果、处理原因和处理记录。 |
| 审计边界 | 操作日志只做审计证据，不能替代异常中心的业务闭环。 |

| 异常类型 | 来源模块 | 处理入口 |
| --- | --- | --- |
| 库存异常 | 团期、库存、出库、盘点 | 异常中心 + 缺货处理 |
| 履约异常 | 出库、司机配送、自提点待自提 | 异常中心 + 履约跟踪 |
| 采购异常 | 采购单、送货单、供应商确认 | 异常中心 + 采购协同 |
| 售后异常 | 售后单、退款失败、责任方争议 | 异常中心 + 售后中心 |
| 资金异常 | 退款、提现、分账、押金不足 | 异常中心 + 对账中心/异常资金 |
| 权限/操作异常 | 越权操作、敏感数据导出、跨范围查询 | 异常中心 + 操作日志 |
| 评价/用户异常 | 低评分、恶意退款、刷券风险 | 异常中心 + 用户与营销 |

| 规则项 | 拍板结论 |
| --- | --- |
| 异常状态 | 状态为待处理、处理中、待外部反馈、已处理、已关闭。 |
| 关联对象 | 异常必须关联业务对象，不能只有文字描述；可关联订单、订单商品、团期、SKU、自提点、供应商、仓库、采购单、出库单、配送任务、售后单、资金流水。 |
| 关联原则 | 能关联多少就关联多少，方便运营从异常直接跳到对应业务页面处理。 |
| 处理动作 | 缺货类可补货、部分退款、替换商品、关闭异常；配送类可重新配送、通知自提点、通知用户、关闭异常；采购类可补货、换供应商、等待供应商反馈、部分履约、关闭异常；售后类可转售后处理、标记责任方、重试退款、关闭异常；资金类可进入对账中心、重试退款/分账、补充账户信息、关闭异常；权限类可冻结账号、收回权限、记录审计、关闭异常。 |
| 关闭规则 | 关闭异常必须填写处理结果和原因；资金、库存、订单履约类异常关闭时必须校验关联单据是否已经处理完成；不能只点“关闭”就结束异常。 |
| 关闭后规则 | 已关闭异常只能查看和追溯，不允许直接编辑处理结果；需要重新处理时重新打开或生成新异常记录。 |
| 驾驶舱展示 | 运营驾驶舱只展示高优先级和超时未处理异常；普通异常留在异常中心列表；驾驶舱异常卡片必须能跳转到异常详情或对应处理页面。 |
| 优先级 | P0 为影响支付、退款、提现、分账、订单履约中断、大面积缺货；P1 为影响部分自提点、部分 SKU、采购延迟、配送延误；P2 为单点问题、低评分、一般操作提醒。 |
| 超时规则 | P0 超过 30 分钟未处理标红；P1 超过 2 小时未处理标黄；P2 当日未处理进入待办提醒。 |
| 边界规则 | 普通售后审核仍在售后中心处理；资金核对仍在对账中心处理；异常中心只承接需要运营介入的问题、差异和失败记录。 |
| 必备字段 | 异常中心必须能看见异常对象、责任模块、发生时间、当前状态、优先级、处理人、处理记录和下一步动作。 |

### 3.2 商品与团期

### 3.2.1 团期发布规则

| 规则项 | 拍板结论 |
| --- | --- |

| 商品与团期关系 | 商品是基础资料，团期是真正可售单元；一个商品可以进入多个团期。 |

| 基础校验 | 校验商品状态、销售城市、销售时间、截团时间、采购仓/履约仓、价格、限购、起售数量、库存售卖模式。 |
| 配送日期校验 | 指定配送日期时，校验配送日期必须大于截团日。 |
| 重叠校验 | 校验同一商品是否存在同时间段售卖的其他团期。 |
| 售后规则校验 | 不校验售后规则，系统使用统计类目配置的默认售后规则。 |
| 自提点范围校验 | 不校验自提点范围，因为第一版不做团期自提点黑白名单。 |
| 缺失项处理 | 有缺失项时不允许发布，只能保存为待发布。 |

| 团期状态 | 修改规则 |
| --- | --- |
| 待发布 | 允许修改团期配置。 |
| 已发布且未截团 | 只能修改商品价格、库存、生产日期、保质期。 |
| 已发布且已截团 | 不允许修改影响下单、采购、库存、履约的数据。 |
| 已生成订单 | 任何修改不能影响已支付订单权益。 |
| 已下架/已过期 | 只允许查看和处理关联异常，不允许重新编辑成可售。 |

| 团期状态 | 说明 |
| --- | --- |
| 待发布 | 配置未发布，可编辑并做发布前校验。 |
| 已发布 | 小程序端按城市和自提点营业状态售卖。 |
| 已下架 | 前台不可继续购买，已生成订单不受影响。 |
| 已过期 | 销售期结束，只允许查看和处理关联异常。 |

### 3.3 交易履约

| 功能 | 详细说明 |
| --- | --- |
| 订单中心 | 按团期、城市、自提点、供应商、仓库、交易状态、支付状态、履约状态查询订单；订单详情展示商品快照、支付流水、退款记录、履约记录、分佣记录和操作日志。 |
| 售后中心 | 售后按订单商品维度发起和审核；支持仅退款、退货退款、补货/补发、部分退款、责任方认定、退款失败重试、分佣扣回和异常联动。 |
| 履约跟踪 | 截团后按团期联动采购、仓库出库、司机配送和司机到点签到；运营后台看结果和异常，仓库后台处理实操任务。 |
| 缺货处理 | 处理发布前库存不足、分拣少件、出库配送异常和用户售后反馈的缺货问题；缺货进入异常中心，可由运营发起补货，采购再生成补货采购单。 |

### 3.3.1 订单与履约规则

| 状态线 | 状态 |
| --- | --- |
| 交易状态 | 待支付、已支付、已取消、已完成 |
| 支付状态 | 未支付、已支付、退款中、已退款、部分退款、退款失败 |
| 履约状态 | 待截团、待采购、待入库、待分拣、待配送、待自提、部分自提、已自提、履约异常 |

| 规则类型 | 详细说明 |
| --- | --- |
| 状态判断 | 后台订单列表可以展示综合状态，但业务规则必须按交易状态、支付状态、履约状态三条线判断，避免取消、退款、发货、自提、分账混在一个状态里。 |
| 下单范围 | 只有已发布且未过期的团期允许下单，用户必须在销售开始时间到截团时间之间下单。 |
| 自提点休假 | 自提点在该团期休假时，小程序端不允许下单；自提点休假只影响新下单，不自动取消已有订单，已有订单必须继续履约或由运营安排转移处理。 |
| 商品快照 | 下单时必须生成商品名、规格、价格、库存售卖模式、团期、生产日期、保质期、供应商、采购仓/履约仓快照。 |
| 未支付取消 | 未支付订单只允许用户取消；运营后台不提供手动取消按钮；15 分钟超时关闭属于系统自动关闭。 |
| 已支付取消 | 已支付未履约订单，运营后台可配置“取消订单”功能权限，取消后进入退款取消流程。 |
| 禁止直接取消 | 已发货、待自提、部分自提订单不允许直接取消；团期下架不影响已支付订单继续履约。 |
| 售后入口 | 订单商品已收货后，用户端即可针对该订单商品发起售后；售后按订单商品维度处理。 |
| 截团汇总 | 截团后系统按团期汇总订单，在仓库后台展示采购单、出库单、司机配送数据。 |
| 虚拟库存履约 | 虚拟库存模式不直接占用仓库库存，截团后根据配送日生成采购单，采购入库后再进入分拣配送。 |
| 仓库库存履约 | 仓库库存模式下单或截团后锁定可售库存，截团后按仓库可售库存生成出库单和司机配送数据。 |
| 防重复采购 | 所有采购单生成时，必须按“同一配送日 + 同一 SKU”扣减已生成或已占用的采购数量，避免重复采购。 |
| 司机签到默认签收 | 司机送达自提点后，司机签到即默认自提点签收，对应订单明细进入待自提。 |
| 签收口径 | 第一版不再单独做自提点强制确认签收，不要求司机或自提点在签收环节提交缺货、破损异常。 |
| 售后追踪 | 用户发现缺货、破损、错货等问题后从用户端发起售后，运营通过售后单、订单明细和配送记录追踪。 |
| 商品级分佣 | 分佣按订单商品维度计算，订单详情表可以先进行商品级分佣。 |
| 最终分账 | 订单主表未完成前不生成最终分账；订单主表已完成后可生成最终分账。 |
| 售后回冲 | 有售后中的订单不影响已生成分账；售后退款完成后，按退款商品和退款金额扣回各方佣金。 |
| 押金抵扣 | 押金是各方余额的一部分但不可提现；只有售后扣回场景允许佣金余额为负，提现不能让佣金余额变负。 |
| 订单后台权限 | 订单取消、退款、导出、改履约信息必须走按钮权限；涉及手机号、地址、财务金额的导出必须记录导出日志。 |

### 3.3.2 售后与退款规则

| 规则类型 | 详细说明 |
| --- | --- |
| 发起维度 | 订单商品已收货后，用户端即可针对该订单商品发起售后；售后按订单商品维度发起，不按整单强制处理。 |
| 售后原因 | 用户可针对少件、破损、变质、错发等原因发起售后，并允许运营后台配置扩展原因。 |
| 并发限制 | 退款待审核、售后审核中、退款中、补货处理中等售后进行中状态，不允许同一订单商品同时再次发起售后。 |
| 再次发起 | 已退款或已关闭售后的订单商品，允许再次发起售后；超出售后期后不允许再发起。 |
| 仅退款 | 缺货、少件、破损、质量问题、过期、无法履约等场景，审核通过后直接进入退款流程。 |
| 退货退款 | 需要退回商品后再退款的场景，必须确认退货后再退款。 |
| 补货/补发 | 可重新履约的缺货、错货、少件，平台直接生成补货单处理；换货第一版不做独立流程，按补货单处理。 |
| 平台补偿 | 第一版暂不做平台补偿。 |
| 售后审核 | 售后必须由运营/客服审核；审核通过和驳回都必须填写原因。 |
| 金额控制 | 退款金额不能超过该订单商品可退金额，支持部分退款。 |
| 责任方 | 涉及供应商责任、仓库责任、司机配送责任、自提点责任的售后，需要标记责任方。 |
| 财务复核 | 高金额退款或批量退款第一版暂不要求财务复核。 |
| 退款关联 | 退款必须关联原支付流水、订单商品和售后单；退款失败进入异常中心，可重试。 |
| 支付状态 | 部分退款后订单支付状态进入部分退款；全额退款后对应订单商品进入已退款。 |
| 佣金扣回 | 退款完成后必须反向扣除相关方佣金，并保留原分佣记录和扣回记录，不能覆盖原记录。 |
| 押金提现 | 押金是各方余额的一部分但不可提现；除非解除合同并手动释放押金，否则押金不允许提走。 |
| 负数边界 | 只有售后扣回场景允许各方佣金余额变为负数；提现不能让佣金余额变为负数。 |
| 资金异常 | 押金不足或佣金倒扣后余额为负时，进入资金异常，后续收益优先冲抵负数。 |
| 退货入库 | 退货退款需要仓库或自提点确认退货后，再进入退款流程；退货商品按实际情况进入可售库存、异常库存或报损，且必须写库存流水。 |
| 不可复售 | 生鲜、临期、破损、过期商品默认不能重新入可售库存。 |
| 异常联动 | 缺货、破损、错货、少件、退款失败、押金不足、责任方争议进入异常中心。 |
| 关联对象 | 售后单必须关联订单、订单商品、团期、SKU、自提点、供应商、仓库、配送任务。 |
| 入口边界 | 异常中心只处理需要运营介入的问题，普通售后审核仍在售后中心处理。 |
| 后台筛选 | 售后中心支持按团期、城市、自提点、供应商、仓库、售后类型、售后状态、责任方筛选。 |
| 后台详情 | 售后中心必须支持查看订单商品快照、支付流水、退款记录、配送记录、分佣扣回记录；五类记录作为售后审核、退款、责任方认定和佣金扣回依据。 |
| 按钮权限 | 审核、退款、驳回、补货、责任方认定必须做按钮权限。 |
| 导出日志 | 售后导出涉及用户、手机号、金额时必须记录导出日志。 |

### 3.4 自提点/团长

| 功能 | 详细说明 |
| --- | --- |
| 自提点管理 | 从原基础资料拆出，维护自提点名称、地址、负责人、城市、营业状态、休假状态、司机签到到店记录、售后反馈、佣金余额和异常记录；自提点停用或团长禁用前必须确认影响范围，已有订单继续履约或由运营安排转移处理。 |
| 团长审核 | 审核用户提交的自提点/团长申请，核对门店地址、联系人、服务能力；通过后生成自提点主体、登录账号、角色和数据权限；团长停用后运营可释放团长粉丝，粉丝可绑定其他团长，后续下单佣金归新团长。 |
| 休假管理 | 审核自提点休假申请，休假通过后影响用户端自提点可选状态、未支付订单继续支付和消息触达；休假团期内不可继续下单，已有订单必须继续履约或由运营安排转移处理。 |
| 签收到店追踪 | 查看司机到点签到、订单明细待自提、售后单和配送记录；第一版不要求自提点强制确认签收。 |
| 团长评分 | 按签收率、缺货率、用户评价、休假频率和履约异常评价团长服务质量，作为整改和激励参考。 |

### 3.5 供应商/采购

| 功能 | 详细说明 |
| --- | --- |
| 供应商管理 | 从原基础资料拆出，维护供应商资料、供货类目、负责人、手机号、登录账号、供货范围、可创建自身 SKU 采购单权限和结算周期。 |
| 采购协同 | 采购单来源必须可追溯，覆盖系统生成、运营发起、采购员创建和供应商创建；供应商创建必须采购审核通过后才转正式采购单。 |
| 送货单 | 送货单必须关联采购单；仓库按送货单收货，支持正常入库、部分入库、拒收入库和异常入库。 |
| 供应商售后 | 处理供应商缺货、延迟送货、入库差异、质量问题、价格异常、退供、索赔和采购异常，影响库存、采购和供应商结算。 |
| 供应商结算 | 按实际入库数量和确认采购价结算，采购价变更、结算导出必须有权限和操作日志，并接入对账中心。 |

### 3.5.1 采购与供应商协同规则

| 来源 | 生成规则 | 风控要求 |
| --- | --- | --- |
| 系统生成 | 虚拟库存团期截团后，根据配送日、SKU、供应商、采购仓生成采购单 | 按同一配送日同一 SKU 扣减已生成或已占用采购数量 |
| 运营发起 | 缺货补货由运营在异常中心发起，采购员生成补货采购单 | 必须关联缺货异常、订单、团期、SKU 和责任环节 |
| 采购员创建 | 采购员可创建手工采购单，来源原因包括团期采购、缺货补货、临时备货、损耗补充 | 必须填写创建原因和采购数量 |
| 供应商创建 | 供应商可为自己名下商品/SKU 主动创建采购单 | 必须提交采购审核，审核通过前不占用采购数量，不能生成送货单 |

| 规则类型 | 拍板规则 |
| --- | --- |
| 仓库库存模式 | 正常不生成采购单，优先生成出库单和司机配送数据。 |
| 采购数量 | 按“配送日 + SKU + 供应商 + 采购仓”汇总；生成采购单时扣减同一配送日同一 SKU 已生成或已占用采购数量。 |
| 补货采购 | 补货采购单必须关联缺货异常、订单、团期、SKU 和责任环节。 |
| 数量调整 | 允许人工调整采购数量，但必须填写调整原因，并记录调整前后数量。 |
| 供应商申请数量 | 供应商创建采购单时只能填写申请采购数量，不能修改平台系统生成的采购数量。 |

| 采购单状态 | 说明 |
| --- | --- |
| 待采购审核 | 供应商创建、采购员创建或补货采购单提交后等待采购审核。 |
| 审核驳回 | 采购审核不通过，采购单作废或退回供应商修改。 |
| 待确认 | 审核通过后进入正式采购单，等待供应商确认。 |
| 已确认 | 供应商已确认采购单，可填写预计送货时间、可供数量、缺货说明。 |
| 待送货 | 采购单已确认，等待供应商生成或执行送货单。 |
| 已送货 | 供应商已送货，送货单进入仓库入库流程。 |
| 部分入库 | 仓库按送货单收货后仅部分商品入库。 |
| 已入库 | 入库完成，更新仓库库存和库存流水。 |
| 已取消 | 采购单取消，必须记录取消原因和操作日志。 |
| 异常 | 供应商缺货、延迟送货、入库差异、质量问题或价格异常。 |

| 供应商创建采购单规则 | 拍板规则 |
| --- | --- |
| 自有范围 | 供应商只能为自己名下商品/SKU 创建采购单。 |
| 默认状态 | 供应商创建的采购单默认状态为“待采购审核”。 |
| 正式采购 | 采购审核通过后才转为正式采购单。 |
| 审核驳回 | 采购审核驳回后，采购单作废或退回供应商修改。 |
| 审核前限制 | 审核通过前不允许生成送货单，不允许进入仓库入库流程，不影响采购数量占用。 |
| 职责隔离 | 供应商不能审核自己的采购单。 |
| 审核留痕 | 采购审核必须记录审核人、审核时间、审核结果、审核意见。 |

| 供应商协同与入库规则 | 拍板规则 |
| --- | --- |
| 数据范围 | 供应商端只能看到自己的采购单、送货单、结算和售后相关数据。 |
| 确认采购 | 供应商需要确认采购单，可填写预计送货时间、可供数量、缺货说明。 |
| 不足量确认 | 供应商确认不足量时，采购单进入异常，运营决定补货、换供应商、退款或继续等待。 |
| 送货单 | 供应商送货时生成送货单，送货单必须关联采购单并进入仓库入库流程。 |
| 入库差异 | 仓库按送货单收货，支持正常入库、部分入库、拒收入库、异常入库；差异必须记录少货、多货、破损、过期、规格不符。 |
| 异常中心 | 供应商缺货、延迟送货、入库差异、质量问题、价格异常进入异常中心，并关联采购单、送货单、供应商、SKU、团期、配送日。 |

### 3.9 风控与审批规则

| 风控对象 | 详细说明 |
| --- | --- |
| 商品发布/团期发布 | 商品作为基础资料，团期作为真正可售单元；发布前必须校验商品状态、销售城市、销售时间、截团时间、采购仓/履约仓、价格、限购、起售数量、库存售卖模式和同商品时间段不重叠；不校验自提点范围和商品级售后规则。 |

| 订单取消 | 未支付订单只允许用户取消，运营后台不提供手动取消按钮；已支付未履约订单可配置“取消订单”功能权限，取消后走退款取消；已发货、待自提、部分自提禁止直接取消。 |
| 售后审核 | 售后必须由运营/客服审核；通过/驳回都必须填写原因，退款金额不能超过该订单商品可退金额，责任方需要标记。 |
| 退货退款 | 必须先确认退货，再发起退款。 |
| 退款操作 | 必须记录退款单、支付流水、操作人、失败原因，失败可重试。 |
| 提现审核 | 不能自动通过，必须财务审核；通过后进入打款流程，驳回释放冻结金额。 |
| 分账入账 | 订单主表已完成后才允许生成最终分账；有售后中的订单不影响已生成分账，售后退款完成后按退款商品和退款金额扣回佣金。 |
| 佣金重算/扣回 | 佣金重算必须保留旧版本并写重算原因；售后佣金扣回必须保留原分佣记录和扣回记录，不能覆盖原记录。 |
| 对账差异 | 支付、退款、提现、分账差异必须进入异常中心，不能只在报表里展示。 |
| 缺货处理 | 必须关联订单、商品、自提点、仓库和责任环节，可触发退款或补发。 |
| 库存售卖模式 | 团期支持配置售卖虚拟库存或售卖仓库库存；虚拟库存用于预售集采并形成采购建议量，仓库库存用于现货售卖并按仓库可售库存控制；两种模式必须在发布前明确配置，发布后不允许切换。 |
| 库存调整 | 报损、盘点差异、手工调整必须审批后生效，并写库存流水；仓库库存售卖影响可售库存，虚拟库存售卖影响虚拟可售量和采购建议量。 |
| 自提点休假 | 自提点只能按团期休假；休假团期内小程序端不允许继续下单；已有订单必须继续履约或由运营安排转移处理。 |
| 团长/自提点禁用 | 必须确认影响范围，不能影响已支付订单履约。 |
| 评论删除 | 必须二次确认，填写原因，保留删除前内容摘要和操作日志。 |
| 优惠券停用 | 未使用券不可再用，已使用订单不受影响，需记录停用原因。 |
| 权限变更 | 角色、菜单、按钮、数据范围变更必须记录操作日志，建议二次确认。 |
| 导出数据 | 涉及用户、订单、财务、手机号时必须受权限控制，并记录导出日志。 |

| 拍板原则 | 详细说明 |
| --- | --- |

| 我的页面 | 我的页面属于个人资产页，进入前必须先登录；展示微信头像昵称、用户订单入口、当前自提点、优惠券、我的收藏、浏览历史、我的评论、商品通知、积分、积分兑换独立入口、提货人信息、团长申请、注销账号等服务入口；签到入口只放在“我的积分”页，不在“我的”页单独展示；“团长申请”为常驻入口，入口内默认申请团长，可选择同时申请团长和自提点；团长审核通过后“我的”页展示团长身份标识，用户具备自提点身份时也展示自提点身份标识，但不展示“我的团长”信息；我的页面入口全部保留，不隐藏到二级页面；站内消息入口展示未读数量红点；积分信息在“我的”页直接露出摘要；用户小程序内提货人手机号、自提点电话、订单快照手机号完整展示，不做脱敏；截图、分享海报和微信分享卡片不展示手机号；注销账号后数据做隔离，仅逻辑删除，不做物理删除。 |
| 我的积分 | 点击“我的”页积分信息进入独立积分页面，展示可用积分、已使用积分、已过期积分、冻结积分、即将过期积分、积分流水、签到入口和积分兑换入口；积分流水类型包含下单待到账、转可用、签到获得、兑换扣减、取消返还、售后扣回、过期等；普通商品支付成功后生成待到账积分，积分比例按运营后台配置，按实付金额向下取整，配送费不参与积分计算；积分有效期从领取时间开始计算，固定为 12 个月后的月底过期；签到页展示连续签到进度、未来奖励和已领取奖励，连续签到奖励由运营人员配置，签到成功后使用轻提示并刷新积分，奖励立即到账为可用积分；积分即将过期提前 3 天通过既定通知渠道提醒一次；积分兑换入口在“我的”页和“我的积分”页同时展示，兑换列表默认按后台配置排序；积分兑换商品需要独立详情页，展示图片、库存、所需积分、预计提货日和兑换规则；积分兑换提交页复用普通结算页样式，但隐藏优惠券、微信支付和实付金额，只展示扣减积分；积分兑换提交前需要再次确认，提交即扣积分并生成订单，不需要待支付状态；积分只用于兑换商品不做下单金额抵扣；积分兑换订单与普通订单区分，订单号前缀不同，兑换商品不使用优惠券、不走微信支付、不产生佣金；积分兑换商品支持每人限兑数量，由后台配置，加购/兑换时校验；积分兑换商品与普通商品库存不共用，由运营后台通过不同模板发布，不展示在小程序首页；兑换提交时积分不足或库存不足，停留在当前页面提示并刷新当前商品状态；积分不足时兑换按钮置灰，并提示差多少积分；兑换提交后立即扣积分并锁库存，扣减时按最早过期积分优先；积分兑换订单未出库前允许取消，取消按钮在订单列表和订单详情底部展示，取消时必须选择取消原因；取消后按原积分批次和原过期时间返还积分，并释放库存；兑换商品不允许售后，仅未出库前可取消，出库后不允许取消；积分兑换空时展示“去首页逛逛”，商品推荐先简单做。 |
| 签到领积分 | 用户每天只能通过“我的积分”页签到入口领取积分；连续签到奖励具体规则由运营人员配置；签到页展示连续签到进度、未来奖励和已领取奖励，并展示连续天数、本笔积分和过期时间；用户当天签到成功后，使用轻提示反馈并刷新积分，签到奖励立即到账为可用积分；连续签到中断后重新计算连续天数，不允许用户补签。 |
| 团长申请 | “我的”页面保留常驻入口，入口名称为“团长申请”；用户小程序“团长申请”页只保留申请表单、审核状态、历史记录和身份标识，不展示佣金、押金、提现规则；用户进入后默认申请团长，也可以选择同时申请团长和自提点，不允许只申请自提点；团长申请必须先登录并授权手机号；团长申请基础字段为姓名、手机号、备注，如果同时申请自提点，则需要填写门店名称和地址；提交后用户端展示审核状态页，状态包括审核中、已通过和已驳回；审核中不允许用户修改申请资料；已有待审核的团点申请时不允许再次提交申请；申请被驳回后允许用户修改资料后重新提交；申请提交后进入运营后台审核，审核通过或驳回时同时发送公众号通知和站内消息；团长审核通过后，用户可登录后续独立建设的团长小程序；团长被运营后台停用后，用户端不再显示团长身份；运营可释放该团长粉丝，粉丝释放后可绑定其他团长，后续下单佣金归新团长；团长申请记录展示历史申请记录，存在多条记录时按时间倒序排列。 |
| 订单列表 | 订单列表属于个人资产页，进入前必须先登录；默认进入时展示“全部订单”，按全部、待付款、已付款、待提货、已完成、售后退款等状态查看订单；“售后退款”入口放在“我的订单”的一个 Tab 中，沿用当前界面已有入口；用户端订单 Tab 保留“待提货”；订单列表不展示提货日，不保留“售后/查看售后”入口，也不显示售后简短标签；订单列表保留“再次购买”入口；用户在订单列表看到“退款中”时，点击订单直接进入订单详情；微信支付结果未回调时订单展示“支付处理中”；积分兑换订单放在普通订单列表里按订单类型展示，订单号前缀使用 JFDH，未出库前订单列表展示取消兑换入口；支付超时、超时支付或跨团支付自动取消后展示为“已取消”并保留“再次购买”入口；用户点击“再次购买”时，如果部分商品已失效或当前自提点不可买，先进入结算页提示具体不可购买商品及原因，然后删除对应商品；再次购买保留原订单商品数量，但需要重新做限购校验；再次购买遇到多规格商品时，默认使用原 SKU；未支付订单取消后点击“再次购买”直接进入结算页；部分商品售后退款后原订单保持原履约状态；订单主状态为“已完成”时仍只在“已完成”Tab 展示，不额外标记；售后单从售后退款列表页查询数据；所有订单默认只展示最近 6 个月内的数据，历史订单查询入口放在订单列表底部，可查询一年内的历史订单记录；订单空时展示“去首页逛逛”，商品推荐先简单做。 |
| 订单详情 | 展示商品、金额或扣减积分、优惠、支付、配送、自提点信息快照、时间线、订单备注、订单类型和提货人订单快照；订单备注为空时隐藏备注栏；订单详情里的订单号支持一键复制，并轻提示“已复制”；订单详情保留“联系自提点”按钮；积分兑换订单展示扣减积分，不展示微信支付、优惠券或实付金额；待支付订单详情提示“库存已锁定，超时将自动取消”；订单详情里的支付、退款、分账、售后规则说明改成状态化展示，只在对应状态出现对应提示；退款中订单详情展示“预计到账时间以微信支付为准”；用户订单不展示提货码或二维码；订单详情中在每个商品上展示对应提货日，提货日延期时展示新的预计提货日并通过公众号和站内消息通知用户；订单整体可显示“待提货”，商品行单独展示“已提货/待提货”；同一订单多个商品不同提货日时，允许用户对单个商品分批确认收货；商品行在价格旁显示数量，取消/退货数量显示在价格行下方，确认收货、申请售后、评论在价格行右侧操作；真正“申请售后”只放在订单详情的商品行；订单详情商品行不展示售后单状态，存在售后中的商品不显示“申请售后”按钮；订单详情底部按钮按订单状态动态展示，积分兑换订单未出库前底部展示取消兑换按钮，取消时必须选择取消原因；售后期结束后订单详情隐藏“申请售后”按钮，状态变更为已完结；退款成功后订单详情商品行展示“已退款”并保留原商品价格，且已退款商品行状态不影响当前商品原有履约状态；退款失败后订单详情商品行展示“退款失败”，允许用户查看售后详情但不允许再次操作；订单详情商品行如果处于补货中，隐藏“申请售后”按钮，避免同一商品并发售后；撤销售后后如果仍在售后期内，订单详情恢复“申请售后”按钮；已完成订单申请售后后，订单主状态保持“已完成”；自提点标记少件或错件后生成补货订单，用户端可在原订单详情展示补货记录；补货/补发单用户端只允许查看，不允许操作；退款成功后展示“原路退回，到账时间以微信支付为准”；用户填写的订单备注在订单详情展示，同时运营后台可见；用户端保留确认收货按钮。 |
| 自动完成 | 用户提货后可自行点击确认收货，点击确认收货时按商品行确认；超过预计提货日后用户未提货，仍允许继续提货；用户未确认时 3 天后自动确认收货；最后一个商品交付后 3 天系统自动完成订单，订单详情不展示“预计自动完成时间”。 |
| 订单核销 | 当前阶段不考虑核销订单，用户端不展示提货码、二维码或核销码，自提点端不做核销订单列表或核销确认流程。 |
| 订单完成与分账 | 订单商品已收货后即可申请售后；订单主表已完成后生成最终分账；分佣按订单商品维度计算。团长佣金只计算粉丝普通商品订单和售后退款订单，积分兑换订单、补货订单不参与团长佣金计算；订单支付成功后按订单商品行生成待结算团长佣金和待结算流水，订单完成后才生成正式佣金和结算流水，佣金流水和佣金明细均按订单商品行生成；订单完成后佣金立即进入可提现范围，团长佣金状态区分待结算、可提现、已提现和押金状态；押金是各方余额的一部分但不可提现，提现不能让佣金余额变为负数；只有售后扣回场景允许佣金余额变为负数；售后中的订单不影响已生成分账，售后退款完成后按退款商品和退款金额扣回团长、自提点、供应商等相关方佣金；押金不足或佣金倒扣后余额为负时进入资金异常，后续收益优先冲抵负数；订单取消或全额退款后，团长佣金直接取消；团长停用后历史已完成订单佣金保留，未结算佣金继续结算；粉丝释放前已下单但未完成的订单，佣金归原团长，订单内记录佣金归属团长，下单后不再实时查询粉丝绑定关系；粉丝释放后可绑定其他团长，后续佣金归新团长；团长佣金提现需要设置最低提现金额，并在提现时告知每笔提现手续费；团长佣金提现失败后，只在团长端展示，用户端不需要显示这类数据。 |
| 手动取消 | 未支付订单支持用户手动取消且不需要填写取消原因，未支付订单取消后点击“再次购买”直接进入结算页；已支付未发货订单允许用户取消但必须选择取消原因，取消原因由运营后台配置，允许选择“其他”并填写备注；取消后直接发起退款并进入退款中，不需要运营后台审核，同时生成售后订单数据；用户取消已支付未发货订单后，订单展示为“退款中”，订单详情展示“预计到账时间以微信支付为准”，底部隐藏“再次购买”，退款成功后展示为“已付款取消”并恢复“再次购买”；已出库订单不允许用户取消并隐藏取消入口。 |
| 售后申请 | 用户端发起售后时，用户端只显示“退货退款”；用户选择“少件/缺货”时，仍提交普通售后申请，由运营后台判断是仅退款或退货退款；运营后台把用户售后改成“仅退款”后，用户端售后详情展示“仅退款”；订单商品已收货后即可针对该订单商品提交售后，并支持按数量发起售后；用户发起售后时，“申请数量”默认选择该商品可申请的最大数量；退款金额按申请数量乘以售价自动计算，用户端不允许修改，只有运营后台审核时可以修改退款金额；售后申请支持填写售后备注，其他说明类内容统一填写在备注中；真正“申请售后”只放在订单详情的商品行，订单列表不保留“售后/查看售后”入口；售后期结束后订单详情隐藏“申请售后”按钮，状态变更为已完结；售后单完结后用户能查看售后详情，但不能再操作；售后原因包含少件/缺货、破损、变质、错发，并允许运营后台配置扩展；退款待审核、售后审核中、退款中、补货处理中等售后进行中状态，不允许同一订单商品同时再次发起售后；已退款、已关闭、已驳回或已撤销后，仍在售后期内允许再次发起售后；用户提交售后后，在运营审核前允许撤销，撤销按钮放在售后详情底部并悬浮展示“撤销申请”；用户点击后需要弹窗二次确认并填写撤销原因，审核后不允许用户撤销；撤销售后成功后停留在售后详情页面，并显示“已撤销”；撤销售后后，售后单在售后记录中展示为“已撤销”，订单详情恢复“申请售后”按钮；用户端不需要单独的“售后记录”列表页，只能从“售后退款”入口进入；售后退款 Tab 默认展示全部售后单，角标只显示未完结售后单数量；售后退款列表展示售后单号、商品图、商品名、售后状态、退款金额和申请时间，退款失败售后单置顶展示；售后记录列表展示审核中、待退货、退款中、退款完成、退款失败、已驳回、已撤销、已关闭等状态；售后详情与订单详情不是同一个页面；售后图片凭证必填且最多上传 6 张，图片凭证必须上传成功后用户才能提交售后申请；售后备注必填并限制 200 个字以内；售后申请提交成功后进入售后详情页面，并展示预计处理时间，例如“运营将在 24 小时内处理”；售后详情展示申请提交、运营审核、待退货、退款中、退款完成/失败、补货处理中等进度节点；售后详情展示“查看原订单”按钮，点击后返回原订单详情；退款金额拆分仅在仅退款、退货退款类型中展示，统一展示商品金额、优惠券影响、实退金额和退回积分；售后单完结后用户可查看售后详情但不能再操作；用户撤销、超时关闭、运营驳回后，售后详情不允许查看历史，只展示当前结果和必要原因；售后单被驳回后，用户允许查看售后详情和驳回原因；售后超时关闭后，用户能看到关闭原因，例如“超时未退货，售后已关闭”；待退货后用户端不提供任何操作按钮，状态展示“等待司机到自提点取回退货商品”并展示退货自提点地址；补货/补发单只在原订单详情展示补货记录，不进入售后退款 Tab，用户端只允许查看，不允许操作；同一商品多次售后后，售后详情页不展示该商品的历史售后记录入口。 |
| 商品评价 | 只允许订单完成后评价；订单完成后不提醒用户评价；已完成订单如果有多个商品，每个商品单独评价；评价支持评分、文字和图片，图片选填；商品详情评价默认按置顶评论优先，其余按最新评价倒序；用户提交低评分评价时先正常展示，运营后台可删除或置顶，不做前置审核；评价图片当前阶段不做审核，运营后台可删除评价；不做匿名评价，不做商家回复；评分、文字内容和图片展示到商品详情买家评价；用户可删除自己的评价，用户删除评价后不允许再次评价，删除后评价状态仍视为已评价，数据层面做逻辑删除；不支持追评和修改评价；运营后台可以删除或置顶评论；订单和评价不展示用户头像昵称，降低隐私风险。 |

| 采购单 | 系统生成、运营发起、采购员创建、供应商创建 | 供应商创建、采购员创建、运营补货采购单必须采购审核；供应商创建审核通过前不占用采购数量，不生成送货单 | 采购单生成 → 采购审核 → 供应商确认采购 → 供应商生成送货单 → 仓库按送货单入库 | 形成采购入库需求 |
| 入库单 | 采购单入库、仓库其他入库、司机退货入库 | 采购入库无需审核；其他入库需仓库审核；退货入库无需审核 | 指定库区 → 接单 → 扫码 → 质检 → 上架 → 完成 | 库存增加 |
| 配送送货单 | 仓库发起、订单自动生成 | 仓库发起需仓库审核；订单自动生成无需审核 | 配送单生成 → 波次下放 → 分拣 → 缺货/报损 → 出库 → 配送 → 自提点交接 | 配送完成 |
| 出库单 | 仓库其他出库、配送单波次下放 | 其他出库需仓库审核；波次下放无需审核 | 出库单生成 → 扫码装车 → 出库/部分出库 → 扣减库存 → 完成 | 库存扣减 |
| 配送退货单 | 退货订单自动生成 | 无需审核 | 退货单生成 → 司机取回 → 回仓交接 → 生成退货入库单 | 合格入库，不合格报损 |

### 7.2 采购管理

| 功能 | 详细说明 |
| --- | --- |
| 人工新增采购单 | 采购员选择供应商、仓库、商品规格、采购数量、采购价和到仓时间，必须填写创建原因，提交后进入待采购审核。 |

| 运营补货采购单 | 缺货补货由运营在异常中心发起，采购员生成补货采购单，必须关联缺货异常、订单、团期、SKU 和责任环节。 |

## 8. 系统任务与中间件

| 任务/事件 | 触发点 | 处理逻辑 | 中间件/组件 |
| --- | --- | --- | --- |

| 未支付取消 | 订单创建后 15 分钟未支付 | 自动取消订单，释放锁定库存，通知用户 | XXL-Job、MQ |
| 支付提醒 | 未支付订单最后 5 分钟 | 公众号提醒用户完成支付 | XXL-Job、公众号通知 |
| 配送数据生成 | 到达配送数据生成节点 | 按配送日、自提点、线路、商品汇总配送数据 | XXL-Job |
| 优惠券事件 | 领取、使用、退回优惠券 | 更新优惠券库存、用户券状态和订单优惠 | RocketMQ |
| 优惠券过期 | 用户券过期未使用 | 标记用户券过期，不退回优惠券库存 | XXL-Job |
| 积分发放 | 订单支付/完成、用户签到 | 普通商品支付成功后按运营配置比例生成待到账积分，实付金额向下取整且配送费不参与；签到奖励按运营配置立即到账为可用积分，写入积分流水并更新用户积分账户 | RocketMQ、MySQL |
| 售后退货履约 | 售后审核需要退货且到达截团/返仓节点 | 根据返仓日期生成退货单，司机到自提点取回退货商品并回仓交接；审核通过且需要退货时发送站内消息和公众号通知；超过运营配置退货时限仍未退货时关闭售后 | XXL-Job、配送系统、公众号通知 |
| 补货订单处理 | 自提点标记少件或错件 | 运营生成补货订单，原订单详情展示补货记录；补货订单不需要用户再次支付，也不产生积分和佣金；第二天无法补货时继续延期补货 | MySQL、配送系统 |
| 积分过期 | 到达积分流水过期时间 | 积分领取后 12 个月后的月底过期，扫描到期且未消耗完的积分批次，生成过期扣减流水 | XXL-Job |
| 订单完成分账 | 订单完成或售后结束 | 生成佣金明细、分账流水，更新分账方账户 | RocketMQ |
| 分账重试 | 分账失败或入账失败 | 重试分账任务并记录失败原因 | RocketMQ、XXL-Job |
| 导出任务 | 后台导出报表/订单/日志 | 异步生成 Excel 文件并提供下载 | XXL-Job |
| 日志记录 | 所有关键按钮操作 | 写业务日志、操作日志、库存流水或审核日志 | MySQL |

## 9. 核心状态清单

### 9.1 订单状态

| 状态线 | 状态 | 说明 |
| --- | --- | --- |
| 交易状态 | 待支付、已支付、已取消、已完成 | 判断订单主流程是否可继续交易；订单商品已收货后允许售后，订单主表已完成后才允许最终分账。 |
| 支付状态 | 未支付、已支付、退款中、已退款、部分退款、退款失败 | 判断支付流水、退款链路和财务风控动作。 |
| 履约状态 | 待截团、待采购、待入库、待分拣、待配送、待自提、部分自提、已自提、履约异常 | 判断采购、仓库、司机配送、司机签到默认签收和异常处理节点。 |

后台订单列表可以展示综合状态，但业务规则必须按三条状态线判断，避免取消、退款、发货、自提、分账混在一个状态里。

### 9.2 售后状态

| 状态 | 说明 |
| --- | --- |
| 待审核 | 用户提交售后，等待后台或相关角色审核。 |
| 已审核/待退货 | 退货退款审核通过，等待用户或自提点退货。 |
| 已审核/待退款 | 仅退款审核通过，等待退款执行。 |
| 已退货/待退款 | 仓库或自提点已确认退货，等待退款。 |
| 退款成功/已审核 | 仅退款场景退款成功。 |
| 退款失败/已审核 | 仅退款场景退款失败。 |
| 退款成功/已退货 | 退货退款场景退款成功。 |
| 退款失败/已退货 | 退货退款场景退款失败。 |
| 已撤销 | 用户在运营审核前主动撤销售后，售后单保留为已撤销。 |
| 已驳回 | 运营驳回售后且必须填写驳回原因，驳回原因展示给用户。 |
| 超时关闭 | 审核通过且需要退货后，用户超过运营后台配置时限仍未退货，系统关闭售后。 |
