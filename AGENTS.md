## 核心业务流程（全员必须理解）

### 主流程：商品到交付

```
商品建档 → SKU/规格绑定 → 商品审核 → 创建团期 → 配置销售/截团/配送
    → 发布校验 → 团期发布 → 用户端可售
    → 用户浏览/加购/下单/支付 → 截团
    → 生成采购/出库/配送数据 → WMS 出库配送
    → 司机送达签到 → 自提点签收 → 用户自提
    → 订单完成 → 分账/佣金 → 提现
```

### 售后流程

```
用户发起售后（按订单商品维度）→ 售后审核
    → 仅退款 / 退货退款
    → 退款执行 → 退款成功 → 佣金扣回
    → 失败则进入异常中心
```

### 关键状态机

| 实体 | 状态流转 |
|---|---|
| 团期 | 待发布 → 已发布 → 已下架 → 已过期 |
| 订单 | 待付款(10) → 已支付(20) → 拣货中(30) → 配送中(40) → 待自提(50) → 已完成(60) / 已取消(70) |
| 订单商品履约 | 待履约 → 待出库 → 配送中 → 待自提 → 已自提 |
| 售后 | 待审核 → 处理中 → 已退款 / 已关闭 |
| 出库单 | 待下放 → 已下放 → 拣货中 → 复检中 → 已装车 → 已出库 |
| 提现 | 待审核 → 已审核 → 打款中 → 打款成功 / 打款失败 |

## 全员红线（任何 Agent 不得违反）

### 业务安全

- 不允许超卖（库存不足必须拦截）
- 不允许重复支付/重复分账
- 不允许绕过状态机修改订单
- 不允许退款金额大于实付金额
- 不允许负数库存
- 不允许截团后继续下单
- 不允许出库完成前司机打卡发车
- 不允许订单完成前做最终分账
- 不允许异常中心绕过原业务模块直接改库存/资金/订单状态

### 数据安全

- 不允许 SQL 拼接，必须参数化
- 不允许密码/密钥/AK-SK 硬编码
- 不允许接口返回明文敏感信息
- 不允许用生产数据做测试

### 代码质量

- 不允许 Controller 里写业务逻辑
- 不允许 Service 里直接写 SQL
- 不允许跨服务直接访问数据库
- 不允许在循环里查数据库
- 不允许魔法数字
- 不允许 Catch 异常后吞掉
- 不允许大事务
- 不允许在 `Sql/mall.sql` 之外执行 DDL

### 资金安全

- 佣金提现不能跳过财务审核
- 团长账户不能透支提现
- 退款不能绕过风控规则
- 分账流水必须可追溯、可对账
- 押金不可提现，只有解除合同手动释放后才允许提走

## MQ 事件规范

- **Topic 命名**：`MALL_模块_业务`（大写），如 `MALL_ORDER_CREATED`
- **消费者组**：`mall_模块_group`，如 `mall_trade_group`
- 消息必须带：msgId、traceId、业务唯一键
- 消费必须幂等，失败重试 3 次后进死信队列

### 核心事件清单

| 场景 | Topic | 一致性策略 |
|---|---|---|
| 商品变更 | MALL_PRODUCT_CHANGED | 本地事务 + MQ 最终一致 |
| 团期发布 | MALL_SALE_PERIOD_PUBLISHED | 本地事务 + 发布前校验 |
| 下单锁库存 | MALL_ORDER_CREATED | Redis 原子锁定 + 订单本地事务 |
| 支付成功 | MALL_PAYMENT_SUCCESS | 支付流水幂等 + 订单状态事件 |
| 未支付超时取消 | MALL_ORDER_PAY_TIMEOUT | 延迟消息/XXL-Job + 幂等取消 |
| 截团 | MALL_SALE_PERIOD_CLOSED | 定时任务 + 批次幂等 |
| 出库完成 | MALL_OUTBOUND_COMPLETED | WMS 本地事务 + FIFO + 库存流水 |
| 售后通过 | MALL_AFTERSALE_APPROVED | 售后本地事务 + 退款事件 |
| 退款成功 | MALL_REFUND_SUCCESS | 退款流水幂等 + 佣金扣回事件 |
| 订单完成 | MALL_ORDER_COMPLETED | 订单状态机 + 分账事件 |

## 修改后浏览器自动验证规范（强制执行）

### 触发条件

以下场景完成后，**必须使用内置沙箱浏览器（preview_url 工具）预览验证**，不打开用户本机的真实浏览器：

| 场景 | 触发者 | 验证内容 |
|---|---|---|
| 前端页面修改 | @Dev | 沙箱中打开修改的页面，检查渲染、交互是否正常 |
| 接口修改/新增 | @Dev | 沙箱中通过 Swagger UI 或 localhost 验证接口可用性 |
| UI/原型调整 | @UI | 沙箱中打开 HTML 原型/前端页面，确认视觉效果 |
| Bug 修复 | @Dev | 沙箱中打开受影响页面，复现修复前的场景验证已修复 |
| 路由/权限变更 | @Dev | 沙箱中打开后台，验证菜单/路由/权限是否生效 |
| 配置文件修改 | @Ops | 沙箱中打开对应页面验证服务是否正常启动 |

### 验证方式

**使用 `preview_url` 工具在沙箱内置浏览器中预览，不调用任何打开本机浏览器的命令。**

```
# 正确：使用沙箱预览工具
preview_url("http://localhost:端口/路径")
preview_url("/绝对路径/index.html")   # 本地静态 HTML 直接预览

# 错误：不允许使用以下命令打开本机浏览器
❌ start chrome "http://..."
❌ start "" "C:\Program Files\Google\Chrome\Application\chrome.exe" "..."
❌ open "http://..."（macOS）
❌ xdg-open "http://..."（Linux）
```

### 验证检查清单

Agent 打开浏览器后，必须确认以下要点：

- [ ] 页面能正常加载（无 404、500、白屏）
- [ ] 控制台无红色报错（F12 → Console）
- [ ] 修改的功能点按预期工作
- [ ] 未修改的页面/功能无回归异常
- [ ] 接口返回数据结构正确（F12 → Network → Response）
- [ ] 样式无错位、布局无塌陷

### 验证结果记录

验证完成后，在回复中附带验证结果：

```
✅ 浏览器验证通过
  - 访问地址：http://localhost:xxxx/xxx
  - 验证点1：xxx 正常
  - 验证点2：xxx 正常

❌ 浏览器验证失败
  - 访问地址：http://localhost:xxxx/xxx
  - 问题：xxx
  - 已自动回滚 / 等待修复确认
```

### 约束细则

1. **不跳过**：只要触发了上述场景，就必须打开浏览器验证，不允许"我认为没问题就跳过"
2. **不等用户催**：Agent 修改完成后主动打开浏览器，不需要用户额外提醒
3. **失败即止损**：如果验证不通过，立即定位问题并修复，不允许带病交付
4. **前后对比**：如果是 Bug 修复，先复现原 Bug 再确认修复效果
5. **服务未启动则启动**：如果项目服务未运行，Agent 应先启动服务，再用沙箱预览工具验证

## Agent 间的铁律

1. **各司其职**：PM 不写代码，Dev 不改架构，QA 只测不改，Ops 不改业务逻辑
2. **文档驱动**：所有决策留痕，所有变更有记录
3. **不越界**：遇到职责边界问题，先问对应 Agent，不自行决定
4. **先读文档再动手**：任何 Agent 接到任务，先读核心输入文档相关章节
5. **暂缓功能绝对不做**：第一版暂缓清单里的功能，不出现入口、路由、接口、权限
6. **SQL 只进不出**：所有 DDL 只追加到 `Sql/mall.sql`，同步 `changelog.md`
7. **高风险动作留痕**：发布、下架、退款、分账、提现、库存调整、导出必须审计
8. **先 compile 再 test**：代码提交前必须 `mvn compile` → `mvn test`
9. **改完必验证（沙箱模式）**：任何代码/页面/接口/配置修改完成后，**必须使用 `preview_url` 沙箱工具预览验证**，绝对不允许调用 `start chrome` 等命令打开用户本机真实浏览器，也不允许仅凭逻辑推断就交付。验证不通过则继续修复，直到验证通过为止。详见上方「修改后浏览器自动验证规范」。
10. **先亮明身份**：任何 Agent 被触发后，回复的第一行必须声明当前角色身份（编号 + 角色名 + 触发方式），让用户一眼知道"现在是谁在说话"。多角色协作时每次发言前都要声明，交接时标注下一棒。详见上方「角色身份声明规范」。
