### 1. 先读文档，再写代码

接到任务时：

5. 设计表结构
6. 写接口、实现、测试

### 2. 代码分层（严格按框架文档执行）

每个业务服务内部按以下结构，不混层：

```
src/main/java/com/mall/{domain}/
├── {Domain}Application.java
├── controller/        ← 入口层，只做参数校验、权限、Result 返回
├── service/           ← 业务逻辑唯一归属
├── domain/            ← 领域模型、状态机、领域规则
├── mapper/            ← MyBatis Plus Mapper
├── convert/           ← MapStruct 转换（Entity ↔ DTO/VO）
├── dto/               ← 出参 DTO
├── vo/                ← 入参 VO
├── event/             ← RocketMQ 事件对象
├── enums/             ← 模块内枚举
├── config/            ← 模块配置
└── job/               ← XXL-Job 任务处理器（如有）
```

资源目录：
```
src/main/resources/
├── mapper/            ← MyBatis XML（复杂 SQL）
├── application.yml
├── application-dev.yml
└── bootstrap.yml
```

> **注意**：`dto/` 放出参，`vo/` 放入参，`convert/` 用 MapStruct 做对象转换，不要用手写 getter/setter 复制。

### 3. 数据库设计规范

- 主键 `id`：bigint，雪花 ID 或自增
- 必备字段：`id`、`create_time`、`update_time`、`is_deleted`（逻辑删除）
- 状态字段：tinyint，并在注释中说明含义，必须有对应枚举常量
- 金额：`DECIMAL(16,4)`，禁止 FLOAT/DOUBLE
- 数量：根据精度用 int / bigint / decimal
- 时间：`datetime` 或 `timestamp`，不要存字符串
- 索引：必须写明设计理由，单表索引不超过 5 个
- 大字段：text/blob 单独拆表
- **表命名按表前缀规范**（见表前缀章节），不用 `t_模块_业务` 格式
- 字段名：下划线命名，全小写，不超过 30 字符
- **所有 DDL 只能追加到 `Sql/mall.sql`，并同步 `Sql/changelog.md`**，禁止在其他地方执行 DDL

### 4. 表前缀规范（严格遵守）

| 分组 | 表前缀 | 归属模块 |
|---|---|---|
| 系统权限 | sys_ | mall-system |
| 用户与主体 | usr_ | mall-user |

| 订单交易 | ord_ | mall-trade |
| 支付退款 | pay_ | mall-payment |

| 供应商采购 | sup_、pur_ | mall-supplier |

| 消息通知 | msg_ | mall-notify |
| 报表导出 | rpt_ | mall-report |
| 搜索同步 | search_ | mall-search |

### 5. MQ 事件规范

- **Topic 命名**：`MALL_模块_业务`（大写），如 `MALL_ORDER_CREATED`、`MALL_PAYMENT_SUCCESS`
- **消费者组**：`mall_模块_group`，如 `mall_trade_group`、`mall_finance_group`
- 消息必须带：msgId、traceId、业务唯一键
- 消费必须幂等，幂等键用业务唯一键
- 失败重试 3 次，之后进死信队列
- 核心状态不得静默丢事件

**核心事件清单（按框架文档）：**

| 场景 | Topic | 一致性策略 |
|---|---|---|
| 商品变更 | MALL_PRODUCT_CHANGED | 本地事务 + MQ 最终一致 |
| 团期发布 | MALL_SALE_PERIOD_PUBLISHED | 本地事务 + 发布前校验 |
| 下单锁库存 | MALL_ORDER_CREATED | Redis 原子锁定 + 订单本地事务 |
| 支付成功 | MALL_PAYMENT_SUCCESS | 支付流水幂等 + 订单状态事件 |
| 未支付超时取消 | MALL_ORDER_PAY_TIMEOUT | 延迟消息/XXL-Job + 幂等取消 |
| 截团 | MALL_SALE_PERIOD_CLOSED | 定时任务 + 批次幂等 |
| 出库完成 | MALL_OUTBOUND_COMPLETED | WMS 本地事务 + FIFO + 库存流水 |
| 售后通过 | MALL_AFTERSALE_APPROVED | 售后本地事务 + 退款事件 |
| 退款成功 | MALL_REFUND_SUCCESS | 退款流水幂等 + 佣金扣回事件 |
| 订单完成 | MALL_ORDER_COMPLETED | 订单状态机 + 分账事件 |

### 6. 接口设计规范

- 统一返回：`{ code: 0, msg: "ok", data: {}, traceId: "xxx" }`
- 接口路径：`/api/{service}/{module}/{action}`
- 幂等：修改接口必须带 `idempotentKey`，由调用方生成
- 分页：统一 `pageNum`（从1开始）、`pageSize`，返回 `total` + `list`
- 时间：出参用 `Long` 毫秒值或 ISO 8601
- 脱敏：手机号、身份证、地址、银行卡必须脱敏（第一版查看/导出受权限控制，并记录操作日志）
- 文档：用 Swagger / Knife4j 自动生成
- 异常：统一用 BusinessException，带错误码

### 7. 枚举与常量规范

```java
// 订单状态（tinyint，状态值和注释必须一致）
public enum OrderStatusEnum {
    PENDING_PAYMENT(10, "待付款"),
    PAID(20, "已支付"),
    PICKING(30, "拣货中"),
    DELIVERING(40, "配送中"),
    PENDING_PICKUP(50, "待自提"),
    COMPLETED(60, "已完成"),
    CANCELLED(70, "已取消");

    private final int code;
    private final String desc;
}
```

- 枚举必须有 code 和 desc
- 数据库存 code，展示用 desc
- 状态流转必须写状态机校验，不能随意改

### 8. 并发与一致性

- 库存操作：Redisson 分布式锁 + 数据库乐观锁（version 字段）
- 订单创建：唯一索引（order_no） + 幂等键 + 状态机校验
- 支付回调：幂等 + 状态机 + 对账日志
- 优先用 **MQ 最终一致性**，避免分布式事务
- 缓存：先写数据库，再删缓存；金额/库存不准只依赖缓存

### 9. 缓存使用规范

- 缓存模式：Cache-Aside
- 更新：先更新数据库，再删缓存
- 穿透：空值缓存（短过期时间）
- 击穿：热点数据永不过期 + Redisson 互斥锁
- 雪崩：随机过期时间
- 严禁：缓存金额、库存等强一致性数据作为唯一数据源
- Redis 异常：核心写链路限流保护，不直接放开库存；读链路允许降级 DB

### 10. 日志与异常

- 每个接口入口打印：入参、traceId、userId
- 每个接口出口打印：出参、耗时
- 异常必须打印 ERROR 日志，包含上下文
- 关键业务操作必须写审计日志（谁、何时、改了什么、改前值、改后值）
- 高风险动作（发布、下架、退款、分账、提现、库存调整、导出）必须审计
- 异常处理：
  ```java
  try {
      // 业务逻辑
  } catch (BusinessException e) {
      log.warn("业务异常：{}", e.getMessage());
      throw e;
  } catch (Exception e) {
      log.error("系统异常，traceId={}，param={}", traceId, param, e);
      throw new SystemException("系统繁忙，请稍后重试");
  }
  ```

### 11. 测试要求

- 单元测试：核心 Service 方法必须覆盖
- 接口测试：用 Swagger / Postman 导出
- 并发测试：库存、订单、支付、波次必须压测
- 测试断言必须覆盖：正常流程、异常流程、边界值
- 提交代码前必须 `mvn compile`，再 `mvn test`
- 测试环境数据必须脱敏

## 红线（绝对不能碰）

### 数据安全
- 🚫 不允许 SQL 拼接，必须参数化
- 🚫 不允许把密码、密钥、AK/SK 硬编码到代码
- 🚫 不允许接口返回明文敏感信息
- 🚫 不允许用生产数据做测试

### 业务安全
- 🚫 不允许超卖
- 🚫 不允许重复支付/重复分账
- 🚫 不允许绕过状态机修改订单
- 🚫 不允许退款金额大于实付金额
- 🚫 不允许负数库存
- 🚫 不允许截团后继续下单
- 🚫 不允许出库完成前司机打卡发车
- 🚫 不允许订单完成前做最终分账
- 🚫 不允许异常中心绕过原业务模块直接改库存、资金、订单状态

### 代码质量
- 🚫 不允许 Controller 里写业务逻辑
- 🚫 不允许 Service 里直接写 SQL
- 🚫 不允许跨服务直接访问数据库（只能通过 API 或 MQ）
- 🚫 不允许在循环里查数据库/调外部接口
- 🚫 不允许魔法数字
- 🚫 不允许 Catch 异常后吞掉不打印
- 🚫 不允许大事务
